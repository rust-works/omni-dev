# ADR-0055: Daemon-Held Read Token for the Non-Polling (Webhook) PR-Status Source

## Status

✅ Accepted (2026-07-20; implemented in #1384, building on the #1378 spike —
[docs/plan/webhook-buffer-spike.md](../plan/webhook-buffer-spike.md)). **Amends
[ADR-0040](adr-0040.md)** ("no secret persisted") for the worktrees service only.

## Context

The worktrees daemon service resolves each worktree's PR/CI badge by polling
GitHub's GraphQL API ([ADR-0053](adr-0053.md)). That is cost-optimal but not
free-in-latency: a badge changes only when the next poll runs. #1378's spike
([docs/plan/webhook-buffer-spike.md](../plan/webhook-buffer-spike.md)) proved a
zero-GitHub-cost, real-time alternative for repos **you own**: GitHub webhooks →
a small Cloudflare Worker + KV buffer (`deploy/webhook-buffer/`) → the daemon
**pulls** the buffered deliveries and reconstructs the same verdict
([src/pr_status/webhook.rs](../../src/pr_status.rs)).

Running that pull requires the daemon to authenticate to the buffer with a
**read/pull token**. Holding — and persisting — a secret is a departure from
[ADR-0040](adr-0040.md), which recorded that the worktrees service "persists
nothing / no secret." The other daemon services either hold no secret
(worktrees, sessions) or hold one only in memory (Snowflake). This ADR records
why the departure is acceptable and how it is bounded, so the trust boundary
stays legible.

The webhook **HMAC secret** is a separate secret, used by GitHub to sign
deliveries and by the Worker to verify them. It must **not** reach the daemon.

## Decision

1. **The daemon holds only a read token, never the HMAC secret.** The HMAC secret
   lives solely as a `wrangler secret` in the Worker, which verifies every
   delivery's `X-Hub-Signature-256` before storing. The daemon can read buffered
   events but can neither forge nor verify deliveries — it never needs the HMAC
   secret, so it never has it.

2. **The read token is persisted `0600` in the daemon data dir.** Resolved from
   `OMNI_DEV_WEBHOOK_READ_TOKEN` (or `_PATH` to a file) via `Settings::get_env_var`,
   it is written to `<data-dir>/omni-dev/webhook.token` with the same
   `ensure_dir_0700` + `write_file_0600` primitives and posture as
   `bridge.token` ([ADR-0039](adr-0039.md) runtime paths). Persisting it keeps it
   out of the process table and out of `settings.json`.

3. **The pull is outbound-only.** The daemon opens an outbound HTTPS connection to
   the Worker; it exposes **no** new local listener or port. [ADR-0036](adr-0036.md)'s
   loopback trust model (nothing on the machine is reachable from off-box) is
   preserved — this feature adds no inbound surface at all.

4. **No secret is logged.** The token rides the `Authorization: Bearer` header,
   which the request log redacts by default ([ADR-0043](adr-0043.md)); the daemon
   never logs the token or the buffer response bodies by default.

5. **Owned repos only.** Installing a webhook needs repo admin, so the fast path
   covers only repos you own. Every other repo keeps working under the retained
   reconcile poll (the GraphQL path), so nothing degrades — non-owned repos are
   simply not real-time.

## Consequences

- The worktrees service now persists exactly one secret (`webhook.token`), and
  only when the `webhook` source is configured. [ADR-0040](adr-0040.md)'s "no
  secret" statement is amended to "no secret **except** the webhook read token,
  under the `0600` posture of this ADR."
- The blast radius of a leaked read token is bounded to **reading buffered CI
  events** for the owned repos whose webhooks point at that buffer — no write
  access, no GitHub credential, no ability to forge deliveries (that needs the
  HMAC secret, which never leaves the Worker).
- The buffer is operator-deployed and opt-in; the default source remains `poll`,
  which is byte-identical to a pre-#1384 daemon and holds no secret.
- A daemon offline beyond the buffer's KV retention (default 3 days) self-heals
  from the reconcile poll rather than showing stale badges, so token/buffer
  outages degrade gracefully to today's behavior.

## Alternatives considered

- **Keep the token in memory only (no persistence).** Rejected: the daemon is a
  long-lived, service-manager-activated process; re-resolving the token from env
  on every restart would force it into the launchd/systemd unit or the process
  environment, which is a *worse* exposure than a `0600` file in the data dir.
- **Give the daemon the HMAC secret and let it verify deliveries itself
  (skip the Worker).** Rejected: that would require an inbound public endpoint on
  the daemon (breaking [ADR-0036](adr-0036.md)) and would hand the daemon a
  forge-capable secret. The Worker exists precisely to keep the HMAC secret and
  the inbound surface off the machine.
- **A per-repo tri-state (poll | webhook | off) instead of a global mode.**
  Rejected for the mode selection: it cannot cleanly satisfy "hide the poll-volume
  UI when the non-polling source is active." The global VS Code setting selects
  the system source; which specific owned repos are webhook-backed is orthogonal
  (the buffer only holds events for repos with the webhook installed).
