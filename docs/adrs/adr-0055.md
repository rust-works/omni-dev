# ADR-0055: Batch Merge-Queue Enqueue over the Daemon, Gated by Server-Side Eligibility

## Status

✅ Accepted (2026-07-24)

## Context

[ADR-0040](adr-0040.md) introduced the `worktrees` daemon service, [ADR-0048](adr-0048.md)
grew it into a live cross-window tree, and [ADR-0049](adr-0049.md) added its first
**destructive** op — `close`, which deletes a linked worktree's files and closes
windows. Every capability so far acts on **local** state: it reads git and the
registry, spawns `code`, or removes a directory.

The natural next action on a finished worktree is **merging** it. With GitHub
merge queue enabled on a repo, merging means *enqueuing*, and doing that per-PR in
the browser across a fleet of worktrees is exactly the tedium the daemon's
cross-window view exists to remove. The value is **batch enqueue with automatic
eligibility filtering**: select N worktrees, act once, and only the ones genuinely
ready-to-merge get queued.

Three facts shaped the design:

1. **This is the service's first op that mutates state *outside* the machine.**
   `close` deletes local files; enqueue changes a pull request on GitHub. That is a
   different *kind* of side effect, not merely a bigger one — it is visible to other
   people, it consumes the shared GitHub API budget, and it cannot be undone by the
   daemon. It is also, on every other axis, **less** dangerous than `close`: no
   local file is created, modified, or deleted.
2. **"Ready to merge" is a compound, server-side fact.** It spans local git state
   (clean, committed, pushed) and remote PR state (an open, non-draft,
   conflict-free, CI-green PR). A client cannot be trusted to compute it — a tree
   row can be seconds stale — and a *partly* correct check is worse than none,
   because it enqueues the wrong PR.
3. **A batch must degrade per-item, never all-or-nothing.** One ineligible or
   un-enqueuable worktree in a selection of ten must not fail the other nine, and
   must not be silently dropped either — the user needs to know *which* and *why*.

The ADR-0040/0039 constraints still bind: reuse the `0600`-socket/`0700`-dir
transport and trust model unchanged, persist no secret and no state, keep the
engine's `Mutex`-never-across-`.await` invariant, and stay wire-compatible with
older clients through the `#[serde(skip_serializing_if = …)]` convention.

## Decision

Add a single **`merge-queue`** op to the `worktrees` service: two-phase like
`close`, daemon-re-validated, but **batched** over a set of paths, with every
eligibility gate evaluated server-side and every enqueue routed through the user's
own `gh`.

### 1. One batched op, not a client-side fan-out

Payload: `{ paths: [...], requester_key?, check?: bool, confirmed?: bool }`.

`close` fans a multi-select out into **one op per target**, because each close is a
largely independent wait on a different window's heartbeat. Merge-queue is the
opposite shape: the dominant cost is a **single** `gh api graphql` round-trip that
can resolve *every* selected branch at once (the same aliasing trick that makes the
badge poll cost 1 point for ~50 branches — see [ADR-0053](adr-0053.md)). Fanning
out N ops would turn one query into N. So the batch is the unit:

- **Phase 1 — check** (`check:true`, or any un-`confirmed` request). Side-effect-free.
  Returns `{ eligible: [{path, number, url, branch}], skipped: [{path, kind, detail}] }`.
- **Phase 2 — execute** (`confirmed:true`). Re-runs the whole evaluation, then
  enqueues. Returns `{ queued, skipped, failed }`.

**A batch always confirms**, carrying over ADR-0049 §1's rule for the same reason:
one gesture now mutates N pull requests, and the modal is the only place the user
sees the full set. Unlike `close`, there is no "no dialog when nothing is at risk"
case — enqueue is a remote, externally-visible mutation, so it always asks.

### 2. Seven gates, evaluated server-side, cheapest first

A worktree is enqueued only if **all** hold. Each failure yields a stable machine
`kind` slug plus a human `detail`, mirroring `close`'s `Note` shape:

| # | Gate | Skip `kind` |
|---|---|---|
| 1 | Clean working tree | `dirty` / `untracked` |
| 2 | Has a commit (non-unborn HEAD) | `no-commits` |
| 3 | Fully pushed (`head == upstream`, `ahead == 0`) | `unpushed` / `no-upstream` |
| 4 | An open PR heads the branch | `no-pr` |
| 5 | The PR is not a draft | `draft` |
| 6 | The PR is not conflicting (`mergeStateStatus`) | `conflicting` |
| 7 | The PR's checks are green | `checks-failing` |
| — | The PR head matches the local head | `stale` |

Gates 1–3 are **local git** and run first, per path, on a blocking thread; only
survivors reach GitHub. A dirty or unpushed worktree is therefore skipped with a
precise reason at **zero** API cost — which also means a selection of mostly-dirty
worktrees is nearly free. A detached HEAD (`detached`) and a non-GitHub remote
(`no-github`) are refused locally for the same reason.

Gates 5–7 are deliberately **stricter than the merge queue requires**. GitHub would
run CI itself, and a `BLOCKED` PR can legitimately enter a queue; we gate anyway,
because the value of the batch action is that the user does not inspect each PR —
so an enqueue that will predictably bounce is better reported as a skip, with a
reason, than accepted and silently rejected later. `BLOCKED` and `UNKNOWN` merge
states are *not* treated as conflicts (the queue resolves those); only `CONFLICTING`
and `DIRTY` are.

The `stale` gate (remote PR head ≠ local head) is the guard against a stale UI: it
refuses to enqueue a PR whose head is not the commit the user is looking at.

### 3. The daemon re-validates on execute; UI gating is convenience only

Phase 2 recomputes **every** gate from scratch and enqueues only what still passes —
it never trusts the eligible-set the client saw in phase 1. This is the same
never-trust-the-client posture as `close`'s main-working-tree refusal, and it is
what makes the ~seconds between render, check, confirm, and execute safe: a worktree
that became dirty, was force-pushed, or had its PR closed in that window is skipped
at execute time and reported in the reply's `skipped[]`. The extension's menu
gating (`viewItem` must be a github-backed *worktree*) and its modal are
discoverability and consent, not guards.

### 4. A dedicated enqueue-time query, not the hot-path badge fragment

`enqueuePullRequest` needs the PR's GraphQL **node id**, which the badge cache
(ADR-0053) does not carry, and the gates need `mergeStateStatus` / `mergeQueueEntry`,
which it also does not. The obvious move — widen the badge fragment — is **rejected**:
that fragment is polled continuously for every open branch, `mergeStateStatus` is
server-side-expensive and transiently `UNKNOWN`, and widening it would force a
schema change on the persisted badge cache. Instead a separate fragment and resolve
path serve merge-queue only, for the handful of explicitly-selected branches. They
share the `statusCheckRollup` sub-selection and its reducer with the badge fragment,
so the CI verdict cannot drift between the badge a user sees and the gate that
blocks them. Because the op must re-validate anyway, always-fresh is a requirement,
not a cost.

### 5. Enqueue rides the existing `gh` choke point, and fails per-PR

Each enqueue is an `enqueuePullRequest(input:{pullRequestId})` mutation issued
through the same `run_gh` wrapper every other GitHub call uses, so it is counted,
rate-limit-attributed, and request-logged identically — and **no credential enters
the daemon** (ADR-0053's posture, unchanged: the token stays inside `gh`).

Failures are per-item by construction. `gh` exits non-zero on a rejected mutation;
rather than propagate that as an error for the whole batch, the reason is captured
into that PR's `failed[]` entry and the loop continues. Merge queue not enabled on
the repo, a PR that became unmergeable, and insufficient permissions all surface as
readable reasons against the specific PR. A PR already in the queue is **idempotent
success** (`already_queued`), not a failure, and skips the mutation entirely.

Enqueues run **sequentially**. The batch is user-selected and small, each item is
one subprocess, and sequential issuance is naturally gentle on the API budget; a
bounded concurrency cap is a trivial later optimisation, not a v1 requirement.

### 6. Both surfaces, one op

The extension adds **Add to Merge Queue** to the tree's `1_pr` menu group, gated on
a github-backed worktree row, reusing the existing multi-select contract. The CLI
adds `omni-dev worktrees merge-queue <PATH>... [--check] [--yes]` — `--check` prints
the eligibility report and stops. Unlike `close` (socket-only until #1361), the CLI
command ships with the op, so the capability is scriptable and reachable headlessly
from day one; it costs one `insta` help snapshot update.

## Consequences

- **The service's first *outbound* mutation — a new kind of reach, not a bigger
  blast radius.** Before this, a socket writer could spawn `code` (ADR-0048) or
  delete a linked worktree (ADR-0049); now it can also put a PR in a merge queue.
  That is externally visible and not undoable by the daemon, but it is bounded
  exactly as everything else here is: same `0600` socket, same same-user
  requirement, and it can only ever act through the user's **own** `gh`
  credentials — it grants no access the caller did not already have from a shell.
  It creates, modifies, and deletes **no local file**, making it strictly safer
  than `close` on the local axis.
- **Eligibility is a check, not a policy the client enforces.** The daemon computes
  the verdict and re-computes it before acting; the extension only decides how to
  present it. A malformed or outdated client cannot enqueue an ineligible PR.
- **Skipped is loud, never silent.** Every excluded worktree comes back with a
  stable `kind` and a human reason, in both phases — the failure mode this op is
  most likely to hit (a selection where half the worktrees are dirty) is also its
  most informative output.
- **Cheap when it fails, one query when it works.** Local gates short-circuit before
  any network call, and the survivors resolve in a single aliased GraphQL query, so
  the API cost is ~1 point per action regardless of selection size, plus one
  mutation per PR actually enqueued.
- **A precondition the daemon cannot fix.** Merge queue must already be enabled on
  the repo (an admin setting). The op does not attempt to enable it; it surfaces
  GitHub's rejection as a per-PR `failed[]` reason.
- **Stricter than GitHub, by choice.** Gating on draft/conflict/CI means a PR the
  queue would have accepted can be reported as skipped. This is deliberate (§2) and
  is the one decision most likely to warrant revisiting — relaxing a gate is a
  one-line change plus a doc update.
- **Wire-compatible and additive.** `merge-queue` is a new op; an older daemon
  rejects it cleanly via the existing unknown-op error, and every existing op,
  stream, and the tray are byte-for-byte unchanged. No new state and no secret is
  persisted.
- **Still Unix-only**, like the rest of the service and companion.

This supersedes nothing: it extends [ADR-0049](adr-0049.md) — carrying over its
two-phase check→confirm shape and daemon-side re-validation — and builds on
[ADR-0053](adr-0053.md)'s `gh api graphql` resolution. See [ADR-0039](adr-0039.md)
for the daemon framework, [ADR-0040](adr-0040.md) for the service, and
[docs/worktrees-service.md](../worktrees-service.md) for the operator guide and the
full `merge-queue` wire contract.
