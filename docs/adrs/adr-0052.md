# ADR-0052: Cross-Window Claude Code Sessions Tracker as a Daemon Service Fed by Three Feeds

## Status

‚úÖ Accepted (2026-07-15)

## Context

There is no way to see, for the logged-in user and across **every** terminal and
VS Code window, which Claude Code sessions are running right now and what each is
doing (working, idle, or blocked waiting on the user). Each source of truth is
partial:

- A **hook** fires inside one `claude` process and knows only that process's
  session. It cannot enumerate the others.
- A **VS Code window** is sandboxed per extension host ([ADR-0040](adr-0040.md)):
  it sees only its own tabs and terminals, never a sibling window's, and no
  extension can enumerate another extension's webviews.
- The **transcript files** under `~/.claude/projects/**/*.jsonl` are a
  machine-wide, passively-readable trace, but on their own they carry no live
  state and no window association.

A verified feasibility study (issue #1210) reached three conclusions, each
adversarially checked:

| Target | Verdict | Only via |
|---|---|---|
| All running sessions + coarse state | ‚úÖ Yes | Hooks + transcript watching |
| All git repos/worktrees open in VS Code | ‚úÖ Yes (already shipped) | The worktrees companion ([ADR-0040](adr-0040.md)) |
| Sessions embedded in VS Code, cross-window | ‚öÝÔ∏è Partial | Companion per window; per-tab attribution is heuristic-only |

There is **no purely-passive, outside-the-editor** way to get the VS Code facts
reliably ‚Äî the same conclusion [ADR-0040](adr-0040.md) reached for worktrees. The
only architecture that beats the per-window sandbox and the per-process hook is a
**rendezvous point**: a single resident process that each feed reports to, which
aggregates the sightings into one consistent view. omni-dev already runs exactly
such a process ‚Äî the daemon ([ADR-0039](adr-0039.md)) ‚Äî and already hosts a
cross-window rendezvous service for worktrees ([ADR-0040](adr-0040.md)). This
decision adds a fourth service to it rather than building anything new.

This is **distinct from #876** (a history-search MCP over the transcript
directory): that searches *past* conversations; this tracks *currently-running*
sessions and their live state. Both watch the same directory and could later
share a watcher.

Two facts, **verified locally** against the installed `anthropic.claude-code`
extension (v2.1.186), shaped the correlation design:

- The extension launches its agent with `--session-id <sessionId>`, so **the
  hook's `session_id` equals the extension's per-tab key**. The three feeds join
  on `session_id` with no heuristics at the *session* level.
- The extension's `activate()` returns `undefined` ‚Äî there is **no public
  `.exports` API** ‚Äî and a tab's `session_id` is not on the VS Code `Tab` object.
  So a companion cannot bind a *specific tab* to a *specific session*; it can only
  report *how many* Claude tabs/terminals its window has. Per-tab attribution is
  therefore heuristic (by `cwd`), and the session‚Üíwindow join is by `cwd`, not by a
  companion-reported id.

State is **inferred**, not first-class: Anthropic closed the request for
dedicated `AgentWorking`/`AgentAvailable`/‚Ä¶ events as *not planned*
(anthropics/claude-code#43058), and the maintainer calls inference from
`PreToolUse`/`Stop`/`Notification` "unreliable". The design must degrade around
that rather than assume precision.

## Decision

Add a **`sessions` service** to the daemon
([`src/daemon/services/sessions.rs`](../../src/daemon/services/sessions.rs)),
implementing the `DaemonService` trait and registered in the default registry
alongside the browser bridge, Snowflake, and worktrees services. It mirrors those
services' **engine/adapter split**: the engine
([`src/sessions.rs`](../../src/sessions.rs), the crate-root `SessionsRegistry`) is
pure in-memory state behind `std::sync::Mutex`es **never held across an `.await`**;
the adapter routes ops, does the git enrichment, renders the tray/status, and owns
the transcript-watcher task.

### Three feeds, each degrading gracefully

1. **Hooks** (Feed 1, target-1 state). `omni-dev sessions hook` is a stdin sink
   Claude Code runs per hook event; it maps the hook JSON to an `observe`/`end`
   op and fire-and-forgets it to the socket. `omni-dev sessions install-hooks`
   merges the block into `~/.claude/settings.json` idempotently. The sink **never
   blocks or fails a Claude turn**: a missing daemon, a bad payload, or any error
   is swallowed and it always exits 0.
2. **Transcript watcher** (Feed 2, discovery + the thinking window). An
   engine-owned background task scans `~/.claude/projects/**/*.jsonl` for new and
   growing files, so a session that started before the daemon ‚Äî or that is working
   through the ~5‚Äì15s "thinking window" where no hook fires ‚Äî is still tracked. It
   parses **only file presence and growth (size/mtime)**, never the per-line
   schema, which is internal and version-unstable. To avoid announcing hundreds of
   long-dead sessions on first scan, only files touched recently are surfaced.
3. **Companion extension** (Feed 3, VS Code source tagging). The worktrees
   companion ([ADR-0040](adr-0040.md), [`editors/vscode/`](../../editors/vscode/))
   is extended to additionally report *its own window's* Claude embeddings ‚Äî
   editor webview tabs (`viewType` containing `claudeVSCodePanel`) and integrated
   terminals (named like Claude Code) ‚Äî to the sessions service. It reports only
   *counts* plus the window's folders, because the extension cannot expose a tab's
   `session_id`.

### Ops over the existing NDJSON control socket

The feeds talk the daemon's existing wire protocol
([`protocol.rs`](../../src/daemon/protocol.rs)) ‚Äî a `DaemonEnvelope { service:
"sessions", op, payload }` per line:

| Op | Payload | Reply | Meaning |
|---|---|---|---|
| `observe` | `{ session_id, cwd?, transcript_path?, event, model? }` | `{ ok: true }` | Idempotent upsert from a hook/transcript sighting; runs the state machine; refreshes `last_seen`. |
| `end` | `{ session_id, reason? }` | `{ ended: bool }` | `SessionEnd`: marks the session `ended` (reaped shortly after). |
| `window` | `{ key, folders[], tabs, terminals }` | `{ ok: true }` | Companion report of a window's Claude embeddings; refreshes the window's liveness. |
| `window-unregister` | `{ key }` | `{ removed: bool }` | The window closed. |
| `list` | ‚Äî | `{ sessions: [entry, ‚Ä¶] }` | The live set, each with its resolved `source`, sorted by `(repo, session_id)`. |

The issue's separate `state` op was folded into `observe`: carrying the hook
`event` (including the classified `Notification` kind) lets the **engine own the
entire state machine** in one testable place, rather than splitting the transition
logic between the hook sink and the daemon.

### Inferred state machine

`observe` maps the sighting's `event` to a state via `SessionState::for_event`:
`SessionStart ‚Üí starting`; `UserPromptSubmit`/`PreToolUse`/`PostToolUse`/transcript
growth `‚Üí working`; `Stop ‚Üí idle`; a `Notification` classified as a permission
prompt `‚Üí waiting_for_permission`; classified as an idle/input prompt `‚Üí
waiting_for_input`; an unclassified notification or a passively-discovered
transcript leaves the state unchanged. The `waiting_for_*` states are **reliable**
(a `Notification` hook fires them directly); `working`/`idle` is best-effort, with
the transcript-growth backstop for the hook-silent thinking window.

### Liveness by reap-on-read, and the `source` join

Liveness mirrors worktrees: `last_seen` + TTL, reaped **inline on every read**, no
background reaper; the maps are capped (512 sessions, 256 windows) with
evict-longest-silent so ingest stays infallible ([ADR-0040](adr-0040.md), #1140).
Session liveness differs in one way worth stating plainly: a session emits nothing
while idle at the prompt, so its only signal is activity. The session TTL is
therefore generous (5 min), and an idle session that goes quiet longer than that
ages out and re-appears the moment it next does anything ‚Äî the accepted limitation
of a hook-based approach given no dedicated liveness event exists.

A session's `source` is **resolved on read** by joining its `cwd` against the live
window reports: a `cwd` under a window that reports ‚â•1 Claude embedding is tagged
`vscode` (carrying that window's key for a focus action); everything else is
`terminal` ‚Äî meaning "not matched to a reporting VS Code window". The join is a
pure path-prefix compare (no disk I/O under the lock).

### Tray and CLI

`menu()` contributes a "Claude Sessions" submenu with one line per session
(`<name> <glyph> <state>`); a `vscode` session's line is a clickable `focus:` action
that reuses the worktrees launcher, while a `terminal` session (no window to focus)
is a plain label. `omni-dev sessions list [-o json]`
([`src/cli/sessions.rs`](../../src/cli/sessions.rs)) prints the live set as a table
or JSON. Like the other daemon commands it is `#[cfg(unix)]`.

## Consequences

- **No new trust boundary.** Requests ride the daemon's existing `0600` Unix
  socket in its `0700` directory ([ADR-0039](adr-0039.md)); **no secret is
  persisted** (in-memory only). The residual exposure is narrow and bounded by
  socket ownership: anything that can read the socket can enumerate your open
  session **cwds/repos and coarse state**, and anything that can write it can
  inject fake sessions ‚Äî but that already requires being the owning local user.
  The bridge's ([ADR-0036](adr-0036.md)), Snowflake's, and worktrees'
  ([ADR-0040](adr-0040.md)) models are untouched. Hooks are **opt-in** user config;
  `sessions hook` writes nothing except the fire-and-forget socket POST.
- **State is coarse and inferred**, by design. `waiting_for_*` is reliable;
  `working`/`idle` is best-effort. If Anthropic later ships dedicated session-state
  events, they would replace the inference without changing the ops or the schema.
- **Idle sessions age out.** With no liveness event, a session quiet longer than
  the TTL disappears and re-appears on its next activity. This is documented in
  [docs/sessions-service.md](../sessions-service.md) rather than papered over.
- **VS Code attribution is session-level, not tab-level.** The hook‚Üîcompanion join
  on `session_id` is clean, but binding a specific *tab* to a specific *session* is
  heuristic (by `cwd`) because the Claude extension exposes no API ‚Äî one Claude tab
  per window/cwd is unambiguous; several in one cwd are not.
- **The transcript watcher never parses the line schema** ‚Äî only presence and
  growth ‚Äî so it is robust to Claude Code changing its internal transcript format.
  It also cannot decode a transcript's `cwd` from the lossy encoded directory name,
  so a watcher-only session (no hook yet) has no `cwd`/`repo` until a hook enriches
  it; `observe` never clobbers known data with `None`.
- **The companion contract grows by two ops** (`window` / `window-unregister`).
  New optional fields follow the protocol's `#[serde(default, skip_serializing_if =
  ‚Ä¶)]` convention, so an older daemon and a newer companion (or vice versa) stay
  wire-compatible.
- Like the rest of the daemon, the service and CLI are `#[cfg(unix)]`; the hook
  sink and transcript scheme are portable, but the socket transport is Unix-only
  (Windows tracked with the broader daemon work, #1363).

See [ADR-0039](adr-0039.md) for the daemon framework this builds on,
[ADR-0040](adr-0040.md) for the rendezvous pattern and companion this extends, and
[docs/sessions-service.md](../sessions-service.md) for the operator-facing guide,
the hook contract, and the companion's wire contract.
