# ADR-0054: Native Windows Daemon Control Plane on a Per-User Named Pipe with Detached-Spawn Activation

## Status

✅ Accepted (2026-07-20; design agreed docs-first in #1363 — implementation
phased per [docs/plan/windows-daemon-port.md](../plan/windows-daemon-port.md))

## Context

[ADR-0039](adr-0039.md) built the omni-dev daemon on a Unix-domain control
socket: `UnixListener`/`UnixStream` in `src/daemon/server.rs`,
`src/daemon/client.rs`, and `src/daemon/single_instance.rs`, with the exclusive
socket bind doubling as the single-instance lock and launchd/systemd providing
socket activation. That choice made the entire daemon runtime Unix-only — every
daemon module except the deliberately cross-platform `paths`, plus the four
daemon-backed CLI trees (`daemon`, `snowflake`, `worktrees`, `sessions`), is
`#[cfg(unix)]`, and on Windows those subcommands do not exist. The supported Windows story is WSL2 (a real Linux kernel, where
`#[cfg(unix)]` holds and VS Code connects via Remote-WSL). What WSL cannot serve
is a **native** Windows extension host, terminal, or Claude Code session: the
`AF_UNIX` socket does not cross the WSL2 VM boundary, and the companion's socket
client (`editors/vscode/src/socket.ts`) has no `win32` branch — on native
Windows it silently no-ops forever. Issue #1363 tracks closing that gap.

The port must preserve ADR-0039's load-bearing properties: the control channel
is **not a network trust boundary** (it is same-user, local-machine IPC); the
NDJSON wire protocol (`src/daemon/protocol.rs`) is transport-agnostic and must
not change; and Unix behavior must remain byte-identical through the refactor.
One Unix property has no Windows analog at all: there is no per-user
demand-activation primitive for pipes (nothing like a launchd `Sockets` dict or
a systemd `.socket` unit), so the activation model is necessarily different.

Alternatives considered for the transport:

- **Loopback TCP with a bearer token** (the browser bridge's model). Rejected:
  it re-introduces exactly the network trust surface ADR-0039 avoided on Unix —
  a `127.0.0.1` listener is reachable by every local process and, via
  CSRF/DNS-rebinding classes, by browser pages — and it adds a token lifecycle
  (issuance, discovery, rotation) that the Unix path does not have, leaving the
  two platforms with asymmetric security models. Port discovery would also need
  a rendezvous file, replacing one filesystem dependency with another.
- **`AF_UNIX` on Windows** (supported by `afunix.sys` since Windows 10 1803).
  Rejected: `std::os::unix` does not exist on Windows builds, tokio has no
  Windows `AF_UNIX` support, and the ecosystem crates that wrap it are
  synchronous — the same seam work would be needed with a worse foundation.
- **The `interprocess` crate**. Rejected: a new dependency that duplicates
  `tokio::net::windows::named_pipe`, which is already compiled into our
  existing `tokio` (`full` features) dependency; the named-pipe accept loop has
  to be written either way.
- **Embedding the username in the pipe name** (instead of hashing the endpoint
  key). Rejected: per-user uniqueness already falls out of hashing a key that
  contains the user's profile directory, and sourcing a username identically
  from Rust and Node is a gratuitous cross-language mismatch risk.

Alternatives considered for activation:

- **A Windows Service.** Rejected: machine-scoped and admin-installed — the
  wrong shape for a per-user, no-elevation developer tool.
- **A Task Scheduler logon task** (or `Run` registry key). Deferred: it would
  add login auto-start but not demand activation (nothing provides that for
  pipes), at the cost of an install/uninstall/escape-hatch surface. It can be
  layered on later without changing the design below.
- **Detached spawn only** — the model the launcher already uses on Unix systems
  without a service manager (the "other Unix" arm of
  `src/cli/daemon/control.rs`). Chosen.

## Decision

1. **We will generalize the control plane behind a platform transport seam**, a
   new `src/daemon/transport.rs` compiled on all platforms: cfg type aliases
   for the client/server stream types, a cfg-split `Listener` wrapper with an
   `accept()` method, and `connect`/`bind` functions keyed by the socket path.
   The private server framing helpers become generic over
   `AsyncRead + AsyncWrite + Unpin`, which monomorphizes to today's exact Unix
   code. The socket path remains the cross-platform **endpoint key**, so
   `DaemonClient`'s `PathBuf` API — and every CLI, MCP, hook-sink, and
   companion call site built on it — is unchanged.
2. **We will use a per-user named pipe as the Windows transport**:
   `\\.\pipe\omni-dev.daemon.<hash16>`, where `<hash16>` is the first 16 hex
   chars of the SHA-256 of the normalized endpoint key (separators `/`→`\`,
   ASCII-lowercased). The name is a pure function of the key, computed
   identically in Rust (`sha2`, already a dependency) and the companion
   (`node:crypto`), pinned by a shared test vector. A `--socket` value already
   of the form `\\.\pipe\...` is used verbatim (power-user escape hatch).
3. **We will make exclusive first-instance creation the single-instance lock**
   (`first_pipe_instance(true)`), the direct analog of the exclusive bind.
   There is no stale-reclaim path: named pipes are kernel objects that vanish
   with their process, so nothing can go stale and nothing is unlinked on
   shutdown (the unlink becomes Unix-only). On `ERROR_ACCESS_DENIED` at bind,
   the existing ping-probe runs for **diagnostics only** — a live daemon yields
   today's "already running" error; a non-answering holder yields a distinct
   error and is never displaced.
4. **We will activate the daemon by detached spawn only**: `daemon start`
   spawns `daemon run` with
   `creation_flags(DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP)` and stdio
   appended to `daemon.log`, mirroring the Unix `spawn_detached` arm (no
   `setsid` analog is needed — Windows children survive parent exit). There is
   deliberately no login auto-start and no demand activation;
   `socket_activated` is always false on Windows.
5. **We will ship phase 1 on the default pipe DACL**, hardened only by
   `reject_remote_clients(true)` and exclusive first-instance creation, and we
   will state the residual exposure plainly: another local user can open a
   client handle to the pipe (weaker than the Unix `0600`-socket-in-`0700`-dir
   posture), and a local process that pre-creates the deterministic name can
   deny service or impersonate the daemon to clients (no Unix analog — the
   `0700` runtime dir prevents foreign binds). Explicit owner-only
   `SECURITY_ATTRIBUTES` plus client-side server-SID verification are a tracked
   hardening follow-up (the point at which a `windows-sys` dependency becomes
   justified). Until that lands, the Windows daemon must not be run on shared
   multi-user hosts.
6. **We will reconcile paths under a single normalization policy** in a new
   `src/fspath.rs`: `compare_key` (on Windows: `dunce::simplified` to strip
   `\\?\` verbatim prefixes legally, `/`→`\`, whole-path ASCII case-fold; on
   Unix: the identity) with a `starts_with_key` prefix test, and
   `display_path` (case-preserving, de-verbatimed) for client-facing fields.
   The policy applies at exactly the three sites where VS Code `fsPath`, git2
   paths, and Claude-hook `cwd` meet: the worktrees adapter's `canonical()`
   join and the sessions engine's `resolve_source`/`focus_folder`. `\\wsl$`
   UNC paths never compare equal to drive-letter paths — by design, they are
   different mount views. We accept the `dunce` crate for this: naive prefix
   stripping silently corrupts exactly the long or reserved-name paths for
   which `canonicalize` produced the verbatim form.
7. **We will keep genuinely platform-specific mechanisms gated**: launchd and
   systemd socket activation, the umask-guarded `0600`-from-birth bind,
   SIGTERM/SIGHUP handling (`lifecycle.rs` already has the Windows Ctrl-C
   branch), the macOS menu-bar tray, and the Unix umask/signal test harnesses.
   WSL2 (including Remote-WSL) remains the supported path for WSL-side
   worktrees; a native-Windows client never joins a WSL-side daemon.

## Consequences

**Positive.** Native Windows gains the whole daemon surface — worktrees,
sessions, snowflake, the hosted browser bridge, and the companion extension —
with zero new runtime dependencies beyond `dunce` (tokio `full` already ships
named pipes; `sha2` is already in-tree). The wire protocol, `DaemonClient` API,
and all `PathBuf`-keyed call sites survive unchanged, so the port is invisible
to Unix users: the transport seam monomorphizes to the identical Unix code, and
`compare_key` is the identity there. The companion needs only endpoint
resolution (a `win32` data-dir branch and pipe-name derivation) — its NDJSON
client, envelope builders, and graceful-degradation behavior carry over, and
its pure-Node test pattern covers the `win32` branches from any OS.

**Negative.** Until the DACL hardening follow-up lands, the Windows control
plane is weaker than Unix `0600`: cross-user pipe opens and pre-start name
squatting are possible, so shared multi-user Windows hosts are explicitly
unsupported in phase 1. A `windows-latest` test job adds CI wall-clock and a
second platform to keep green. There is no demand activation: after a reboot
(or `daemon stop`) the daemon is down until `daemon start` or the first
`daemon run`. Windows path semantics (case-folding, verbatim prefixes) add a
normalization layer that Unix never needed, with a documented benign edge: two
paths differing only by case inside a case-sensitive NTFS tree would falsely
merge, where today's Windows behavior falsely splits the common case.

**Neutral.** ADR-0039 is **extended, not superseded**: it remains the daemon
architecture record, and this ADR owns the transport generalization and the
Windows platform decisions. When the port ships, ADR-0039's Status gains an
inline amendment (its established convention) pointing here, and the platform
notes in [ADR-0040](adr-0040.md), [ADR-0048](adr-0048.md), and
[ADR-0052](adr-0052.md) are repointed from issue #1363 to this ADR. The
implementation is phased (transport seam → Windows transport → CLI un-gating →
launcher, with companion and path normalization in parallel) per the roadmap in
[docs/plan/windows-daemon-port.md](../plan/windows-daemon-port.md), which also
carries the executable detail — the naming spec and test vector, the exact
signature changes, and the CI cut lines.
