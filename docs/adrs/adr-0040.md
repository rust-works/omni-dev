# ADR-0040: candle + earshot VAD + LocalAgreement-2 as the Latency-Tolerant Streaming ASR Floor

## Status

✅ Accepted (2026-06-10) — builds on [ADR-0033](adr-0033.md) and [ADR-0035](adr-0035.md); complements [ADR-0037](adr-0037.md); the async streaming seam is ADR-0038 (in flight on [#933](https://github.com/rust-works/omni-dev/issues/933))

## Context

[ADR-0033](adr-0033.md) made `candle` the production **batch** ASR runtime under
the pure-Rust / no-native-toolchain framing. Streaming was deferred to the #806
umbrella. The question this ADR answers: **what is the streaming backend that
works everywhere** — including Windows, which has no other real-time ASR path
in this project (the Voxtral FFI tier in [ADR-0037](adr-0037.md) is
non-Windows by design)?

The [#969 spike](https://github.com/rust-works/omni-dev/issues/969) (branch
`issue-969-vad-candle-spike`, `spike-candle-streaming/SPIKE.md`) evaluated
VAD-driven chunking + LocalAgreement-2 over candle Whisper `tiny.en` on the
committed 5-minute monologue fixture, paced at 1×, on an Apple-Silicon CPU:

| Gate                | Result                                          |        |
|---------------------|-------------------------------------------------|--------|
| RTF                 | 0.34 (during-speech ~0.44 → ~2.3× CPU headroom) | ✅ pass |
| WER                 | 9.2 %                                           | ✅ pass |
| Time-to-final       | 0.73 s mean / 1.42 s max                        | ✅ pass |
| Peak RSS            | ~429 MB                                         | ✅ pass |
| Determinism         | 3× byte-identical                               | ✅ pass |
| C++/CMake-free      | pure Rust, no `ort`                             | ✅ pass |
| Partial-latency P95 | **2.41 s**                                      | ❌ vs the strict ≤ 1.0 s *interactive* gate |

**Root cause of the partial-latency failure (structural, not tunable):** candle
Whisper costs ~0.5–0.6 s per inference regardless of window length — the
encoder always runs on a zero-padded ~3000-frame (30 s) mel, and
LocalAgreement-2 re-decodes the whole window every cadence because there is no
streaming KV-cache. That fixed cost forces an irreconcilable trade-off:
throttling partials (`min-window 2 s`) keeps RTF healthy but blacks out the
first ~2 s of every phrase; frequent partials (`min-window 0.5 s`) push
during-speech RTF toward 1 and the pipeline falls behind unboundedly. No
operating point passes both RTF ≤ 0.5 and partial-latency P95 ≤ 1.0 s.

Two measures matter and they are not the same: **average RTF** says whether the
pipeline keeps up over a whole session (silence is free); **partial latency**
says how far the displayed transcript trails the speaker *during speech*. The
spike's configuration passes the first decisively and cannot pass the second.

The spike therefore returned a **CONDITIONAL GO**: viable as a
*latency-tolerant* tier whose display lag is ~1.5 s typical / up to ~3 s,
**bounded and non-drifting** as long as during-speech RTF stays < 1 on the host.

## Decision

We will ship `CandleStreamingTranscriber` (`--backend
whisper-candle-streaming`, [#974](https://github.com/rust-works/omni-dev/issues/974))
as the **latency-tolerant, lowest-common-denominator (LCD) cross-platform
streaming ASR backend** — the universal floor every platform can fall back to,
and the only streaming option on Windows until a streaming-native model lands.

### 1. Algorithm: VAD silence-gating + cadence re-decode + LocalAgreement-2

The production port of the spike-validated state machine, on the synchronous
`Transcriber` seam already on `main`:

- **`earshot` VAD** (`src/voice/vad.rs`) silence-gates the decode window —
  only voiced audio is decoded (the RTF lever) — and provides utterance
  endpoints.
- **Cadence inference** re-decodes the accumulated window every
  `cadence_secs` of new audio, skipped entirely when the window is unchanged
  (the redundant-inference skip — the dominant RTF saving at silence
  endpoints).
- **LocalAgreement-2** commits the longest common word prefix of consecutive
  hypotheses as non-revisable `Final`s; the volatile tail streams as
  `Partial`s. The hard window cap re-decodes the full window before flushing
  (lossless cap). `Final`s carry the real average-logprob confidence from the
  shared `WhisperEngine` (`src/voice/backends/candle.rs`), not the spike's
  hard-coded 1.0.

### 2. Recommended operating envelope (the defaults)

`tiny.en, max-window 5 s, cadence 1.0 s, silence 0.3 s, min-window 2.0 s,
VAD threshold 0.5` — tuned in #969 to **maximise keep-up headroom, not minimise
lag**. These are `StreamingConfig::default()`; the knobs are exposed on the
Rust API only (`CandleStreamingTranscriber::with_config`). `silence_secs` is
the one knob expected to need per-deployment tuning (0.5 s cuts more
conservatively); `silence ≥ 0.8 s` is known-bad (windows collapse into the
cap mid-speech: WER 35 %, RSS 601 MB in the sweep).

The keep-up contingency is part of the contract: the bounded-lag property
holds only while **during-speech RTF < 1** on the deploying host
(Apple-Silicon measured ~0.44, i.e. ~2.3× slower-CPU headroom). The
model-gated envelope test plus the `voice-streaming-keepup` CI workflow
(Linux + Windows runners) check this.

### 3. Non-goal: interactive sub-second latency

Sub-second partial latency is structurally unreachable on candle Whisper (see
Context) and is **out of scope for this backend permanently**. Low-latency
interactive streaming requires a streaming-native model (causal encoder +
KV-cache) and stays with the Voxtral work.

### 4. Backend tiering

| Tier                       | Backend                                          | Platforms                          | Latency class             |
|----------------------------|--------------------------------------------------|------------------------------------|---------------------------|
| Universal floor (LCD)      | `whisper-candle-streaming` (this ADR)            | all, **including Windows**         | bounded ~1.5–3 s lag      |
| Low-latency native         | Voxtral pure-C FFI ([ADR-0037](adr-0037.md), #933) | Linux / Apple-Silicon (non-Windows) | interactive               |
| Low-latency pure-Rust      | candle Voxtral Realtime (#936, future)           | all, including Windows             | interactive (target)      |

On Windows, interactive use rides this LCD tier (~2 s lag) until #936 lands.

### 5. Rejected approaches (do not re-walk)

- **Silero VAD via `candle-onnx`** — candle-onnx 0.10 *parses and executes*
  the Silero v5 graph without error, but the output is **numerically wrong**:
  a near-constant ~0.0005 for silence, a loud 200 Hz sine, and real speech
  alike (expected 0…0.95). Op presence in candle-onnx's match table does not
  guarantee correct execution. Evidence: `spike-candle-streaming/src/bin/vad_probe.rs`
  on the spike branch. `earshot` (pure-Rust WebRTC-style GMM VAD,
  **zero dependencies** — `cargo tree -p earshot` shows none, and the
  pre-existing `cc`/`cmake` edges via `aws-lc-sys`/`libgit2-sys`/`onig_sys`
  are unchanged) is the working replacement.
- **Mel-cropping to cut encoder cost** — cropping the mel to real content
  frames instead of candle's fixed ~3000-frame zero pad **breaks candle
  Whisper**: it fails to emit EOT and hallucinates repeated tokens (spike
  measured RTF 4.5, WER 1210 %). The encoder cost is fixed; accept it.
- **`base.en`** — ~2× slower per inference; strictly worsens the binding
  keep-up/latency trade-off and only improves the already-passing WER.

## Consequences

### Positive

- Every platform — critically Windows — has a working streaming ASR backend
  with zero native-toolchain requirements, reusing the already-installed
  `whisper-tiny.en` model and the existing batch inference code
  (`WhisperEngine`).
- Bounded, non-drifting display lag with quantified CPU headroom; the
  envelope is regression-tested against the #969 baseline (RTF 0.34 /
  WER 9.2 % / time-to-final 0.73–1.42 s / RSS ~429 MB).
- Deterministic output (greedy decode, pluggable `UlidRng`) keeps the
  streaming path snapshot-testable.

### Negative

- ~1.5–3 s transcript lag is user-visible; this tier is wrong for
  interactive UX and must not be marketed as such (`docs/asr-backends.md`
  carries the caveat).
- Keep-up is host-dependent: a CPU ≳2.3× slower than the measured
  Apple-Silicon baseline erodes the bounded-lag guarantee. Weak hardware
  degrades to unbounded lag rather than failing loudly.
- Every cadence re-decode pays the full fixed encoder cost — the design
  spends CPU to buy simplicity and portability.

### Neutral

- The backend targets the synchronous `Transcriber` seam on `main`; when the
  async seam (ADR-0038, #933) merges, an async adapter wraps the same state
  machine — a follow-up, not a blocker.
- The spike harness (`spike-candle-streaming/`, sweep scripts, `vad_probe.rs`)
  stays archived on `issue-969-vad-candle-spike` per the #826 convention; the
  production port carries only the validated algorithm.

## Out of Scope (Tracked Elsewhere)

- Low-latency interactive streaming — #936 (pure-Rust Voxtral Realtime) and
  the Voxtral FFI path (#933, ADR-0037).
- The async `StreamingTranscriber`/`AsyncAudioInput` seam itself — ADR-0038
  on #933.
- Diarisation/word-level alignment for streaming events — the decode window
  holds voiced-only samples, so `Final`/`Partial` ranges are
  segment-granularity by design.
