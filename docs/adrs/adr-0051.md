# ADR-0051: Do Not Pre-Trust Daemon-Opened Worktree Windows; Recommend Parent-Folder Trust

## Status

✅ Accepted (2026-07-14)

## Context

The `worktrees` daemon service opens VS Code windows by spawning the editor CLI on
a folder — `code <folder>` — from three places: the tray focus action
([ADR-0040](adr-0040.md)), the tree view's double-click, and the socket `open` op
([ADR-0048](adr-0048.md)). When the target is a folder VS Code has **never seen
before**, the new window starts in **Restricted Mode** behind VS Code's
[Workspace Trust](https://code.visualstudio.com/docs/editing/workspaces/workspace-trust)
gate, and the user must click through a trust prompt. The worktrees workflow is
*built around* many short-lived per-branch worktrees, so it hits this prompt
constantly — friction the service is otherwise designed to remove (#1297).

The launcher cannot fix this as written. Trust is enforced by the VS Code
workbench purely from the **folder path** against a **per-user trusted-folders
list**, independent of *how* the folder is opened, so nothing the daemon passes on
the `code` command line changes the outcome in the general case. The options were
investigated:

- **`code --disable-workspace-trust <folder>`** is a real flag (accepted by the
  installed build, exit 0), but hidden from `code --help`. It is **session-only**
  — the docs state it "only affects the current session" — and, crucially, once a
  VS Code instance is already running, `code <folder>` **forwards the open to the
  resident instance** (the CLI returns immediately without spawning a new editor
  process), which never received the switch. The daemon's normal state is "windows
  already open", so the flag is a no-op in exactly our case. This was confirmed
  empirically on VS Code 1.128.0 with other windows open: a fresh, never-trusted
  folder opened with the flag still landed in Restricted Mode (a companion A/B test
  could not reproduce a reliably-trusted window from the flag), while the global
  trust store held no persisted entry for it. An opt-in
  `OMNI_DEV_VSCODE_DISABLE_TRUST` env that passes the flag would therefore *appear*
  to fix the problem while doing nothing in the common path — a false sense of a
  fix.
- **Pre-seeding the global trust store** — the trusted-folders list lives in VS
  Code's global `state.vscdb` → `ItemTable`, key `content.trust.model.key`
  (`WORKSPACE_TRUST_STORAGE_KEY`), value `{ "uriTrustInfo": [ { "uri": …,
  "trusted": true } ] }`. Writing it *does* genuinely trust a path, but the write
  only sticks while VS Code is **fully quit**; a running instance's in-memory cache
  clobbers external writes on its next flush. The schema is internal and
  version-fragile. A background daemon writing another app's private SQLite while
  that app may be running is unsupportable.
- **Enterprise policy** can disable Workspace Trust globally
  (`security.workspace.trust.enabled=false`) but cannot pre-populate a trusted
  *list*, and disabling the feature for the whole machine is a heavier security
  posture change than this friction warrants — not the daemon's call to make.
- **Extension / Workspace Trust API** is read-only and consent-gated
  (`isTrusted`, `onDidGrantWorkspaceTrust`, `requestWorkspaceTrust()` prompts the
  *current* workspace only). There is no "trust folder X" call, and an extension's
  `globalState` cannot reach the internal trust store, so the companion cannot
  pre-trust either.
- **Parent-folder trust** — trusting a folder in VS Code trusts it *and all of its
  subfolders*, persistently. Trusting the worktree parent once (e.g.
  `~/wrk/work-trees`) makes every subfolder, including every daemon-spawned window,
  open trusted thereafter. It is a one-time UI action and the only robust,
  supported route available today. No supported declarative "default trusted
  folders" mechanism exists; the upstream request
  [microsoft/vscode#291933](https://github.com/microsoft/vscode/issues/291933) is
  open with no maintainer solution.

## Decision

**We will not add any launcher-side trust handling. We will document
parent-folder trust as the recommended workaround.**

- The daemon keeps spawning `code <folder>` unchanged — no `--disable-workspace-trust`
  flag, no opt-in env such as `OMNI_DEV_VSCODE_DISABLE_TRUST`, and no write to VS
  Code's trust store. The flag is a no-op in the daemon's normal already-running
  state, and shipping it (even opt-in) would advertise a fix that does not work;
  the store-seed is unsupportable from a resident background process.
- The operator guide [docs/worktrees-service.md](../worktrees-service.md) gains a
  **Workspace Trust** section recommending the one-time parent-folder trust and
  recording, so they are not re-proposed, why the flag and the store-seed are not
  used.
- We will revisit if upstream microsoft/vscode#291933 ships a supported declarative
  mechanism, or if VS Code changes the CLI so a forwarded open can carry a
  per-window trust decision.

## Consequences

- **No fragile or misleading code ships.** The launcher stays a single unchanged
  `code <folder>` spawn (its existing absolute-existing-directory guard and PATH
  resolution untouched), and the daemon never writes another application's private
  storage. There is no opt-in flag that silently does nothing to mislead an
  operator or a future maintainer.
- **The friction is real but resolved once, by the user.** The recommended fix is
  a manual, one-time action (trust the parent folder), not a zero-config default —
  a documented limitation, not a code fix. After it, every daemon-opened worktree
  window opens trusted, which matches the workflow's expectation.
- **The empirical finding is on record.** The `--disable-workspace-trust`
  behaviour with an instance already running (forwarded open, switch dropped,
  window still Restricted) is captured here and in the operator guide, so the
  option does not get re-litigated on the assumption that it "should" work.
- **A supported path may appear later.** If microsoft/vscode#291933 lands, a future
  ADR can supersede this one with a declarative pre-trust; until then this is the
  standing decision.

This extends [ADR-0040](adr-0040.md) and [ADR-0048](adr-0048.md) (the launcher and
`open` op it concerns) and supersedes nothing. See
[docs/worktrees-service.md](../worktrees-service.md) for the operator-facing
guidance.
