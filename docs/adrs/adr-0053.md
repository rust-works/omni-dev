# ADR-0053: Worktree PR Badges Resolved in the Daemon via `gh api graphql`

## Status

✅ Accepted (2026-07-15)

Supersedes [ADR-0050](adr-0050.md), which decided the opposite — resolve badges
entirely extension-side, with no daemon, protocol, or wire change. That record
remains as historical context; the reasoning that changed is set out below.

## Context

[ADR-0050](adr-0050.md) resolved the worktree PR badge in the companion extension:
one `gh pr list --json …,statusCheckRollup` per repo **on repo-expand**, cached
~60 s per `owner/name`, matched to a worktree by head branch. [#1324] then made the
check state a colored `✓`/`✗`/`●` file decoration.

[#1337] found the badge is never refreshed, and can report a confident green `✓` on
a PR whose CI is still running — or has already failed.

The reducer was not at fault; it is almost never *run*. `fetchPrBadges` has exactly
one call site: `getChildren(repoNode)`. Badges are therefore recomputed only when a
repo node's children are rebuilt — on expand, on manual refresh, or on a re-render
driven by a daemon snapshot. **But the snapshot carries worktree topology, not CI.**
While CI runs and no window opens or closes, zero snapshots are pushed, zero
re-renders happen, and nothing re-asks GitHub. The 60 s cache is never even
consulted, because nothing asks. A comment in `treeDataProvider.ts` — *"A re-render
(a new snapshot) re-runs this and re-fetches, keeping both fresh"* — encodes the
wrong assumption that made this invisible.

Four facts, measured live against `rust-works/omni-dev`, reshaped the decision:

1. **ADR-0050's central objection does not apply to a `gh`-based daemon.** It
   rejected daemon-side resolution because it *"would introduce a GitHub token into
   the daemon, directly at odds with the worktrees service's … 'persists nothing /
   no secret' posture"*. That is true of an HTTP client holding a token. It is not
   true of shelling out to `gh`, which holds the credential itself — the same
   reasoning [ADR-0003](adr-0003.md) already applies to every other GitHub read in
   this codebase, and the daemon already shells out to `code` for the tray's focus
   action. The token still never enters the daemon.
2. **Cost is invariant to repo, worktree, and window count** — the premise the
   extension-side decision was optimising against does not hold for the right query.
   `repository(owner:,name:)` and `ref(qualifiedName:)` take no `first:`/`last:`
   argument, so neither is a GraphQL *connection* and neither contributes to cost.
   One aliased query covering **every** (repo, branch) pair costs **1 point** of the
   5,000/hour budget (measured: 1 up to ~50 branches, 2 at 100, 4 at 200).
3. **`gh pr list` cannot express that query, and costs more.** Its `--json
   statusCheckRollup` is an alias for a `commits(last:1)` connection — one request
   per PR, cost 2 — and `--json` cannot emit a cheaper shape at all.
4. **The badge must not be focus-gated.** Frequent window switching is the intended
   workflow and unfocused windows must show live status. Every comparable tool
   (vscode-pull-request-github, gh-dash, GitLens) focus-gates its polling, because
   N windows each polling is untenable. A single coalescing daemon has no such
   constraint — which is precisely what makes the requirement affordable, and is a
   stronger argument for daemon-side resolution than the cost argument #1337
   originally used.

The ADR-0040/0048 constraints still bind: reuse the transport and trust model, add
no persisted secret or state, keep the streamed snapshot cheap, stay best-effort.

## Decision

Resolve PR badges **in the daemon**, from a background poller, and fan the result
out over the existing `subscribe` stream.

### 1. One `gh api graphql` call for everything

A `pr_status` engine (`src/pr_status.rs`) builds a single query aliasing every
`(repo, branch)` pair the tree knows about, runs it through `gh api graphql` on a
blocking thread, reduces the reply, and caches the badges. Cost is 1 point per
poll regardless of scale.

Open PRs are read off the **`Ref`**, filtered to `OPEN` — not off the `Commit`.
`Commit.associatedPullRequests` returns whatever PR *introduced* that commit, which
on `main` is the last **merged** PR from an unrelated branch; it would paint a false
badge. `Ref.associatedPullRequests(states:OPEN)` matches on **head**, reproducing
the extension's rule exactly (verified: `main` is the *base* of many open PRs and
correctly resolves to nothing).

### 2. Poller-owned cadence, adapted to CI, not to budget

The poller (`WorktreesService::start_pr_poller`, mirroring `start_menu_refresh` and
the Snowflake keep-alive heartbeat) polls ~10 s while any badge is **pending** —
matching `gh pr checks --watch`, which uses 10 s when a human is actively watching a
run — doubling to a **30-minute ceiling** once everything is terminal, and polling
nothing at all while no window is registered. `OMNI_DEV_DAEMON_PR_POLL` overrides
the fast end.

The backoff exists for **battery and wakeups**, not quota: at 1 point per poll the
budget never binds. The 30-minute ceiling is the cross-tool consensus for background
PR polling.

The poller reads its targets off the coalescing `TreeSnapshotCache` rather than
walking git again, so it adds no second per-worktree git walk — the idle-CPU cost
[#1305] removed.

### 3. `pr` on the wire, pushed only when a verdict moves

`TreeWorktree` gains `pr: { number, isDraft, checks, url }` behind
`skip_serializing_if`, so an older client stays byte-identical. The poller bumps the
registry change-notify **only when a badge actually changes**; the server's existing
snapshot diff then pushes to every window exactly when CI state moves, and never
otherwise. An unconditional bump would defeat the diff and re-push to every window
on every poll — the cost this design exists to avoid.

`isDraft` is deliberately camelCase, against the payload's snake_case convention:
the extension's `PrBadge` inherited the name from `gh`'s JSON and every consumer
already reads that key.

### 4. The reducer moves to Rust, unchanged

`rollupCheckState`/`checkEntryState` are ported verbatim — failure dominates, then
pending, then success; skipped/neutral pass; **anything unrecognised is pending, so
a still-resolving or unknown check is never a false pass**. Their `github.ts` unit
tests move with them. This makes the move behaviour-preserving; hardening the
reducer against the `needs: gate` fan-out window is separate work.

### 5. The extension degrades, and the setting gates rendering

Against a daemon predating this ADR — which omits `pr` — the extension falls back to
its own rollup-free `gh pr list` and shows the PR number and draft marker **without**
a check glyph: nothing extension-side polls, so a verdict there could not refresh,
and a stale `✓` is worse than none. The provider asks the fallback only for branches
the daemon left unbadged, so against a current daemon it runs no `gh` at all.

`omniDevWorktrees.showPullRequests` previously worked by not running `gh`. Since the
badge now arrives on the snapshot, gating the fetch suppresses nothing, so the
setting strips badges on the way in instead.

## Consequences

- **The daemon's no-secret posture is preserved, not widened.** Because resolution
  shells out to `gh`, no GitHub token enters the daemon; the credential stays where
  the user put it. [ADR-0040](adr-0040.md)'s *"persists nothing / no secret"* and
  [ADR-0043](adr-0043.md)'s *"no secret material is ever persisted or printed"* both
  still hold verbatim. **This is why ADR-0050's central objection is defused rather
  than overridden** — the disagreement is narrower than it looks.
- **What does change is that PR data now crosses the socket.** ADR-0050 noted "no PR
  data ever flows over the socket" as a property of its design. That property is
  given up. The data is the user's own PR state, on a `0600` same-user socket, and
  is not secret — a materially weaker concession than the token ADR-0050 refused.
- **A new hard dependency on `gh` in the daemon.** Previously `gh` was only an
  extension-side dependency. The daemon now resolves it via `OMNI_DEV_GH_BIN` → a
  well-known-path probe → bare `gh`, because launchd hands it a minimal `PATH`
  (the same tactic the tray's `code` launcher and the extension's `resolveGhBin`
  already use). Absent or unauthenticated `gh` omits badges silently; the explicit
  "Open Pull Request…" action still surfaces the real error.
- **Cost is now invariant to window count**, and lower in absolute terms than the
  status quo: one 1-point call per poll versus one 2-point call per repo per window
  per expand.
- **Unfocused windows show live CI state** — the requirement no focus-gated,
  per-window design can meet.
- **Wire-compatible both ways.** A newer daemon's `pr` is ignored by an older
  extension (unknown fields ride along harmlessly); an older daemon's missing `pr`
  triggers the extension's degraded fallback. This matters because the extension
  ships via Open VSX and the daemon via crates.io, so they version independently.
- **Tension with [#1306] is only apparent.** That made ahead/behind lazy because it
  was O(worktrees) and expensive. A PR badge is resolved once for the whole tree in
  one network call, and read from a cache on the snapshot's hot path — a different
  cost class. The badge must ride the snapshot precisely because a lazy op cannot
  push, and pushing is the entire fix.
- **Still GitHub-only and view-only**, and still best-effort: the tree renders fully
  without badges, and a failing poll leaves the last known ones in place rather than
  blanking the rows.
