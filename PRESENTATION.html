Can Parakeet-TDT 0.6B be ported to pure Rust + candle?

GO (conditional on day-1 CPU RTF gate)

Branch: issue-871-parakeet-rust-port-spike · Day 1 of a 2–3 day box · SPIKE.md · COMPARISON.md

The gap that motivated this

PathWER on the 5-min fixtureSource
parakeet-tdt-0.6b-v2 (MLX fork)3.65 %#856
sherpa-onnx-streaming-zipformer-en6.81 %#859
candle + whisper-tiny.en7.13 %#826

~3.5 pp on the table vs the current C++-free path.

Cheap alternatives (whisper-base.en; sherpa-rs via C++) are either unproven or violate ADR-0033's C++-freeness gate.

This spike asks: is there a pure-Rust port of Parakeet that closes the gap?

What we measured — 3 probes

  • Probe 1 — Op coverage. Enumerate every op in conformer.py, attention.py, rnnt.py, parakeet.py; map each to candle.
  • Probe 2 — Weight conversion. MLX safetensors → candle-loadable safetensors. Throwaway converter; the mapping table is the artefact.
  • Probe 3 — Numerical parity. Encoder block-0 FFN1 sub-path: layer-by-layer MAE/max-AE between the fork and a candle stub.

Output: a go/no-go decision. Not the port. Encoder forward-pass parity is enough; TDT decoder + joiner are downstream.

The catch — CPU RTF is the open risk

Parakeet has ~15× the parameter count of whisper-tiny.en

  • 600 M params vs 39 M
  • 2.47 GB vs ~40 MB
  • 24-layer FastConformer (d_model 1024)

candle + whisper-tiny.en is already too slow on CPU

  • #826: RTF 1.3 paced single-thread
  • Falls behind realtime
  • Naive Parakeet on CPU plausibly lands in RTF 15–25

The 0.016 RTF measured in #856 is MLX-on-GPU + unified memory — it does not predict candle-on-CPU. The spike did not measure runtime cost of the ported model.

Streaming-latency reality (from COMPARISON.md)

Even on parakeet-mlx's hardware-advantaged path (MLX/GPU + unified memory):

Sherpa is ~12× faster on streaming at the same cadence — even comparing against GPU-accelerated parakeet.

On the candle/CPU path that the port would land in, the gap will be larger, not smaller.

Recommendation: GO (conditional)

Recommend the candle port — but gate it on a day-1 CPU RTF probe.

  1. Day 1 of feature task: port encoder forward pass only, load weights, run on 5 s mel input, measure single-thread CPU RTF.
  2. If encoder_RTF > 1.5 on single thread: ABORT. Fall back to sherpa-rs (waive ADR-0033) or whisper-base.en.
  3. If encoder_RTF ≤ 1.5: continue with the remaining 10+ days; consider int8 quantisation + multi-thread BLAS as mitigations.

Bounded discovery cost: ~1 day to learn whether the rest of the port is worth doing.

What this unblocks

If the day-1 CPU RTF gate fails, the port degrades to a defer — and (a) or (b) become the answer.

Closing

The spike succeeded in the sense that matters: it produced a decision the project can act on.

  • Op coverage, weight conversion, numerical parity: all clear
  • Effort estimate: bounded at ~2.5 weeks P50
  • Open risk: CPU RTF unmeasured — promoted to a day-1 gate, not buried
  • Cross-runtime context: sherpa is a real alternative if C++-freeness is negotiable
