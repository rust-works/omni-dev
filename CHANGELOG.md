## [0.33.0] - 2026-07-08

### Added
- **Companion VS Code extension for the worktrees service** ([#1111](https://github.com/rust-works/omni-dev/issues/1111)): the daemon's worktrees service (engine, `omni-dev worktrees list`, tray submenu, git enrichment) shipped fully but had **no data producer** — the registry only fills via the `register`/`heartbeat`/`unregister` ops that, per [ADR-0040](docs/adrs/adr-0040.md), are driven exclusively by a "first-party companion VS Code extension (separate deliverable)" that did not exist, so the feature was inert end-to-end (`worktrees list` and the tray were always empty). That extension now ships in [`editors/vscode/`](editors/vscode/): a thin TypeScript per-window reporter that, over the daemon's `0600` Unix control socket (newline-delimited JSON), `register`s the window's open folders under a fresh per-`activate()` UUID, `heartbeat`s every ~10s (re-registering when a restarted in-memory daemon replies `known: false`), re-registers on workspace-folder changes, and `unregister`s on deactivation — and no-ops silently when the daemon is not running. It recomputes the daemon's socket path (`<data_dir>/omni-dev/daemon.sock`, matching the Rust `dirs` crate on macOS and Linux) and reports only raw folder paths, leaving branch/ahead-behind enrichment to the daemon (#1186) so the reporter stays thin. It is kept fully isolated from the Rust build: single-crate `cargo` never descends into `editors/`, which is added to `Cargo.toml`'s `exclude` so `cargo publish` never ships it, and a new path-filtered `vscode-extension.yml` workflow type-checks, unit-tests, builds (esbuild), and packages a `.vsix` artifact without touching any Rust job. Publishing to the VS Code Marketplace / Open VSX is a deferred follow-up (it needs a publisher account and CI secrets); until then, install the CI-built `.vsix` with `code --install-extension`. See [docs/worktrees-service.md](docs/worktrees-service.md) and [ADR-0040](docs/adrs/adr-0040.md).
- **JIRA attachment upload and delete — the last read-only CRUD gap closed** ([#1114](https://github.com/rust-works/omni-dev/issues/1114)): JIRA attachments were pull-only — the client had only `get_attachments` and the CLI/MCP surface exposed only `jira attachment download` / `images`, while Confluence already had the full upload/list/download/delete set. You could copy a file *off* an issue but never push a log/screenshot/artifact *onto* one, nor remove a stale attachment — a routine "attach evidence to a ticket" automation was impossible. Two verbs close it, reusing the existing multipart plumbing. **`jira attachment upload <ISSUE> <FILE>...`** accepts one or more files and streams each from disk (`ReaderStream`, never fully buffered) as repeated `file` parts of a single multipart `POST /rest/api/3/issue/{key}/attachments` with the `X-Atlassian-Token: no-check` XSRF header, printing metadata for every created attachment. **`jira attachment delete <ATTACHMENT_ID>`** issues `DELETE /rest/api/3/attachment/{id}` (permanent — JIRA has no trash/purge) behind the same destructive guard the Confluence delete uses: an interactive `[y/N]` confirm, `--dry-run` to preview without an API call, and `--force` to skip the prompt. Matching MCP tools ship too: **`jira_attachment_upload`** (takes `file_paths`, returns the created attachments as YAML) and **`jira_attachment_delete`** (guarded by `confirm: true`, mirroring the sibling `jira_delete` tool rather than deleting silently). See [docs/user-guide.md](docs/user-guide.md#jira-attachments) and [docs/mcp.md](docs/mcp.md).
- **Non-interactive Snowflake auth: programmatic access tokens and key-pair JWT** ([#1108](https://github.com/rust-works/omni-dev/issues/1108)): external-browser SSO was the *only* implemented Snowflake auth method, so the daemon could not authenticate headless — no CI, servers, or display-less containers (the v2 client that would have brought these was declined in [#1003](https://github.com/rust-works/omni-dev/issues/1003), leaving the capability gap untracked). Two non-interactive methods now sit alongside SSO on the same clean-room **v1** REST client, selected by a new `SNOWFLAKE_AUTHENTICATOR` env var (resolved from the daemon's environment, then `settings.json`, once at startup): **`programmatic_access_token`** (alias `pat`) presents a PAT from `SNOWFLAKE_TOKEN` in place of a password (Snowflake requires the user to be covered by a network policy to use one), and **`snowflake_jwt`** (aliases `keypair`/`key_pair`/`jwt`) signs a short-lived RS256 JWT locally from an unencrypted PKCS#8 private key at `SNOWFLAKE_PRIVATE_KEY_PATH` (or inline `SNOWFLAKE_PRIVATE_KEY`), authenticating against the public key registered via `ALTER USER … SET RSA_PUBLIC_KEY`. Unset or `externalbrowser` keeps the interactive default byte-for-byte unchanged. All three finish through one shared `session/v1/login-request` path and yield the same session tokens, so the bounded session pool, transparent token renewal, and the keep-alive heartbeat are auth-method-agnostic and untouched. JWT signing uses `aws-lc-rs` (already in the tree via rustls — deliberately **not** the `rsa` crate, the RUSTSEC-2023-0071 reason `snowflake-connector-rs` was dropped); passphrase-encrypted keys are not yet supported, and setting `SNOWFLAKE_PRIVATE_KEY_PASSPHRASE` fails fast with decrypt guidance. No secret is persisted (in-memory only, wrapped so it stays out of logs); no CLI-surface or daemon wire-protocol change. See [docs/snowflake-service.md](docs/snowflake-service.md#authentication).
- **`SNOWFLAKE_HOST` override for PrivateLink and custom API hosts** ([#1224](https://github.com/rust-works/omni-dev/issues/1224)): the Snowflake API host was always derived as `<account>.snowflakecomputing.com`, so AWS/Azure PrivateLink endpoints and gov/custom hosts were unreachable — the client already honoured a `host` override (`SnowflakeClientConfig.host` / `api_host()`) but nothing populated it. A new `SNOWFLAKE_HOST` env/`settings.json` key, when set, is used verbatim as the API host, resolving through `SnowflakeEngineConfig::from_env_and_settings` (via a `host_override_from` helper that trims and treats blank as unset) and threading into every pooled session in `create_session_with_base`. Engine-global and read once at daemon startup, consistent with the other config-only settings; unset keeps the derived host unchanged. No CLI-surface change. See [docs/snowflake-service.md](docs/snowflake-service.md#configuration).
- **`snowflake disconnect --id` / `--all` over the socket and CLI** ([#1228](https://github.com/rust-works/omni-dev/issues/1228)): the engine already had `disconnect_all()` and `disconnect_by_id()`, but the daemon socket op and `omni-dev snowflake disconnect` exposed only the exact `{account, user}` pair form — the bulk and by-id variants were reachable only from the macOS tray, so non-macOS operators had to restart the daemon to bulk-evict sessions. The `disconnect` op/subcommand is overloaded with two more mutually-exclusive selectors: `--id <ID>` (a single pooled session) and `--all` (every session), guarded by a required clap `ArgGroup` so exactly one selector is given (account/user still require each other for the pair form). The socket reply now carries a `count` alongside `disconnected`, so existing callers of the by-id/pair reply stay wire-compatible. See [docs/snowflake-service.md](docs/snowflake-service.md).
- **CSV/TSV output and `--out-file` for `snowflake query`** ([#1227](https://github.com/rust-works/omni-dev/issues/1227)): `snowflake query` gains `csv` and `tsv` output formats — RFC 4180-compliant field quoting, so embedded quotes, commas, and newlines are escaped — alongside the existing JSON and YAML, plus an `--out-file <PATH>` flag that writes results to a file instead of stdout. Column ordering is preserved and duplicate column names are disambiguated with a numeric suffix (`SELECT 1, 1` yields a `_2` column), and every format now ends with a trailing newline for consistency. This makes query results directly consumable by spreadsheets and data pipelines while leaving the JSON/YAML workflows unchanged. See [docs/snowflake-service.md](docs/snowflake-service.md).
- **`coverage diff --ignore-filename-regex` to exclude files from the delta** ([#1220](https://github.com/rust-works/omni-dev/issues/1220)): a new repeatable (and comma-separated) `--ignore-filename-regex` flag on `coverage diff` excludes files from the coverage-delta computation, matching the semantics of `cargo llvm-cov --ignore-filename-regex`. Patterns are compiled into a `RegexSet` with upfront syntax validation and applied — after strip-prefix normalisation to repo-relative paths — **symmetrically** to both the head and baseline reports before the diff, so the two share the same denominator and `--fail-under-patch` gates plus indirect-change lists stay valid even when the baseline predates the exclusion rule. Wired through to the MCP `coverage_diff` tool as `ignore_filename_regex` (`serde(default)`). See [docs/coverage.md](docs/coverage.md) and [docs/mcp.md](docs/mcp.md).
- **Worktrees service names the parent repo and marks linked worktrees** ([#1249](https://github.com/rust-works/omni-dev/issues/1249)): `worktrees list`, `daemon status`, and the tray "Worktrees" submenu now identify which repository each open window belongs to. `GitStatus` gains `main_repo` (the parent repository name, derived from git's common dir — handling both the normal `.git`-subdirectory and bare `.git`-suffixed layouts) and an `is_worktree` flag, computed even for unborn/detached HEAD so a worktree always names its parent. A linked worktree now displays its parent repo name (not its folder basename) and is visually distinguished with a fork glyph (`⑂`), where a normal checkout uses `·`; the CLI table and tray labels prefer the daemon-computed `main_repo` over the companion-reported name. The tray also collapses to one clickable line per window (the live stats label doubles as the focus action) instead of a stats row plus a separate focus row. Both new fields are `#[serde(skip_serializing_if)]`, so the wire protocol stays backward-compatible. See [ADR-0040](docs/adrs/adr-0040.md) and [docs/worktrees-service.md](docs/worktrees-service.md).
- **Cross-service request-log correlation for daemon-served requests** ([#1198](https://github.com/rust-works/omni-dev/issues/1198)): when the CLI or MCP calls through the daemon (Snowflake queries, browser-bridge requests), the HTTP records the daemon spawned were logged under the *daemon's* own `invocation_id`, so `omni-dev log --id <cli-invocation>` could not surface both halves of one logical operation. The originating client's `invocation_id` is now threaded across the control socket (a new optional `origin_invocation_id` on `DaemonEnvelope`, set via `with_origin()`) and the browser bridge (a new `X-Omni-Bridge-Origin` header), and `request_log::scope_origin_id()` overrides the `invocation_id` for the duration of the daemon-side work — so daemon-triggered HTTP records now correlate back to the caller while still reading `via_daemon: true` (and preserving `source`). The origin id is non-secret (not a token). Backward compatible: older clients omit the field and the daemon skips correlation. Closes the follow-up split out of [#1122](https://github.com/rust-works/omni-dev/issues/1122). See [docs/log.md](docs/log.md).

### Changed
- **Upgrade `rmcp` from 1.7.0 to 2.1.0** ([#1222](https://github.com/rust-works/omni-dev/issues/1222)): bump the MCP SDK (`rmcp` + `rmcp-macros`) to 2.1.0 and migrate to its new API surface across all ten MCP tool modules and the resource handlers — `Content` → `ContentBlock`, the removed `RawContent` / `RawResource` / `RawResourceTemplate` intermediaries replaced by direct variants, and the `Resource::new()` builder with chainable `with_*()` methods. Internal migration only; the advertised tool/resource surface is unchanged (manifest + lockfile bump).

### Fixed
- **Worktrees tray no longer pegs a core or stalls daemon shutdown** ([#1249](https://github.com/rust-works/omni-dev/issues/1249)): the per-worktree git enrichment added in [#1186](https://github.com/rust-works/omni-dev/issues/1186) ran on the macOS GUI thread on every tray `menu()` call, so its `git2` I/O pegged a core and could stall daemon shutdown. `WorktreesService` now holds the registry behind an `Arc` and caches a menu snapshot rebuilt by a background tokio task every 2s; `menu()` serves the cached snapshot (falling back to an inline compute before the first refresh or outside a tokio runtime), the tray event loop throttles service polling to at most once per second, and `shutdown()` cancels and joins the refresh task. The service is a no-op under unit tests (which run without a runtime). See [ADR-0040](docs/adrs/adr-0040.md).

### CI/CD
- **Path-split gate to accelerate docs-only PRs** ([#609](https://github.com/rust-works/omni-dev/issues/609)): a new `gate` job runs `dorny/paths-filter` on pull requests and, for docs-only / skills-only changes (`*.md`, `docs/**`, `.claude/skills/**`; anything else fails safe to full CI), skips the heavy jobs while still publishing every required status context — so a docs PR merges without waiting on the full matrix and without bypassing branch protection (a skipped *step* inside a passing job reports success, not a skipped *conclusion*). Non-PR events (main pushes, release tags) always take the full path. Centralises the decision in one gate rather than a separate `ci-skip.yml` that would silently rot as the matrix or branch-protection rules change. See [docs/ci-path-split.md](docs/ci-path-split.md) ([#598](https://github.com/rust-works/omni-dev/issues/598)).

## [0.32.0] - 2026-07-06

### Added
- **Per-origin `--allow-origin` allowlist for the browser bridge** ([#1185](https://github.com/rust-works/omni-dev/issues/1185)): `browser bridge serve --allow-origin` (and the daemon's `--bridge-allow-origin`) is now **repeatable** and scoped per connecting tab instead of a single global value. Each value is a bare `ORIGIN` (a tab on that origin may reach it) or a `CONNECT=OUTBOUND` mapping (a tab connecting from `CONNECT` may reach `OUTBOUND`); repeats under one connecting origin accumulate its outbound set. The WebSocket-upgrade gate now admits any connecting origin that is a configured **key**, and the per-request outbound-URL check reads the outbound set bound to the **tab a request is routed to** — so a Grafana tab and a Facebook tab each carry only their own grant and neither can borrow the other's scope. Default-closed is unchanged: with no `--allow-origin`, the gate stays open (the session token is the gate) and outbound is relative-URLs-only. The per-request `request --allow-origin` override still wins for a single request. Trust boundary unchanged; see [ADR-0036](docs/adrs/adr-0036.md) and [docs/browser-bridge.md](docs/browser-bridge.md#outbound-request-scope-default-closed).
- **MCP tools for the newer subsystems — Snowflake, request log, transcript, coverage, git amend, browser bridge — plus `jira_search` convenience filters** ([#1120](https://github.com/rust-works/omni-dev/issues/1120)): the MCP server exposed 91 tools for the older subsystems (git/jira/confluence/atlassian/ai/config/datadog) but **zero** for the newer ones — the combined router in [`src/mcp/server.rs`](src/mcp/server.rs) stopped at eight families, so an agent could not run a Snowflake query, search its own request log, fetch a transcript, diff coverage, apply commit-message amendments, or drive an authenticated bridge request even though each already existed as a CLI command with a reusable, non-printing core. Ten tools across five new routers close the gap, each delegating to that same core (MCP is a transport, not a fork): **`snowflake_query`/`snowflake_sessions`/`snowflake_disconnect`** send `query`/`sessions`/`disconnect` ops over the daemon's Unix control socket (so they require a running daemon, and first-time `(account, user)` auth opens external-browser SSO on the daemon host); **`log_search`** reuses the CLI filter matrix + renderers (`oneline`/`json`/`full`), capturing output instead of streaming (`--follow` has no MCP form); **`transcript_youtube_fetch`/`info`/`list_langs`** call the `TranscriptSource` trait (the mutating `sync` is omitted); **`coverage_diff`** reuses `DiffCommand::run`, returning the rendered report + patch percentage + gate result and never failing the call (reports `below_gate` instead); **`git_amend_commits`** is the deterministic apply-messages-from-YAML counterpart to `git_twiddle_commits`, taking the amendments as an inline YAML string (a new `AmendmentFile::from_yaml_str` + non-printing `run_amend` core) and refusing already-pushed commits unless `allow_pushed` is set; **`browser_bridge_request`** drives one non-streaming request through a running authenticated tab via the shared `BridgeClient::send` (not read-only — a non-GET method can mutate remote state; `serve`/`harvest` remain CLI-only). Separately, the MCP **`jira_search`** tool now accepts the `project`/`assignee`/`status` convenience filters the CLI has (with `jql` made optional) by reusing an extracted `build_jql_from_filters` shared with the CLI; its `fields` parameter remains the documented informational no-op it already was. The `sync`, `serve`, `harvest`, and daemon-lifecycle surfaces stay intentionally CLI-only. See [docs/mcp.md](docs/mcp.md#tool-catalog).
- **`daemon start` auto-starts at login on Linux via a systemd user unit** ([#1174](https://github.com/rust-works/omni-dev/issues/1174)): `omni-dev daemon start` gains login persistence on Linux, mirroring the macOS launchd LaunchAgent (the deferred second half of [#1112](https://github.com/rust-works/omni-dev/issues/1112), whose detach fix already landed). On a host with a systemd **user** manager it now writes a socket-activated `omni-dev-daemon.socket` + `omni-dev-daemon.service` pair under `~/.config/systemd/user/` and runs `systemctl --user enable --now omni-dev-daemon.socket`: systemd owns the control socket (created `0600` via `SocketMode`) and demand-spawns `daemon run` on the first client connect, and `WantedBy=sockets.target` re-activates it at login. `daemon run` adopts the inherited descriptor via a hand-rolled `sd_listen_fds` implementation ([`src/daemon/systemd.rs`](src/daemon/systemd.rs); reads `LISTEN_FDS`/`LISTEN_PID`, no libsystemd dependency, and sets `FD_CLOEXEC` so the socket is not leaked into child processes), reusing the same socket-activation shutdown invariant as macOS (the manager owns the inode, so shutdown does not unlink it). `daemon stop` stops and disables the socket unit — the analogue of launchd's `bootout` — so it stays down. When no user manager is available (containers, non-systemd distros, or the new `OMNI_DEV_DAEMON_DISABLE_SYSTEMD=1` escape hatch) `start` falls back to the existing detached `setsid` spawn with no login auto-start, exactly as before. One Linux-specific wrinkle: because systemd re-arms the socket the instant the service goes inactive, `daemon restart` no longer polls the daemon *down* (a probe `connect` would re-activate it) — it shuts down and lets the relaunch plus readiness ping drive a clean handoff. See [ADR-0039](docs/adrs/adr-0039.md) and [docs/browser-bridge.md](docs/browser-bridge.md#running-under-the-daemon).
- **Global `--model` / `--beta-header` flags and `--ai-backend openai|ollama|bedrock`** ([#1118](https://github.com/rust-works/omni-dev/issues/1118)): `--model` and `--beta-header` are now global flags on the root command (propagated as `OMNI_DEV_MODEL` / `OMNI_DEV_BETA_HEADER`), so the documented invocation shape `omni-dev --model claude-opus-4-6 git commit message twiddle …` — which clap previously rejected outright — parses, and the old post-subcommand placement keeps working via clap's `global = true`. The per-subcommand copies on `ai chat`, `git commit message twiddle|check|staged`, and `git branch create pr` are removed (both flag positions still parse; MCP tools keep their explicit `model` parameters, which outrank the env). `--ai-backend` is extended from `{default, claude-cli}` to `{default, claude-cli, openai, ollama, bedrock}`, so the three backends that were previously reachable only through `USE_OPENAI`/`USE_OLLAMA`/`CLAUDE_CODE_USE_BEDROCK` env vars are now first-class flag values. `OMNI_DEV_MODEL` slots in above the per-family model vars and, like every resolver input, is settable via `~/.omni-dev/settings.json` env bundles/profiles. See [docs/ai-backends.md](docs/ai-backends.md#dispatch-order-and-model-selection).
- **`--set-field` accepts comma-separated values for array-typed fields** ([#1172](https://github.com/rust-works/omni-dev/issues/1172)): array-typed JIRA fields (labels, components, fix versions, multi-selects) previously required YAML flow syntax — `--set-field 'Labels=a,b,c'` YAML-parsed to a scalar string and was rejected with `expected a sequence for array field 'Labels'`, and the error never named the `[a, b, c]` form users needed, so most looped through several syntaxes before landing on it. The shared field resolver now treats a scalar on an array-typed field (schema `type: array`) as comma-separated shorthand: split on `,`, each element whitespace-trimmed, empty elements dropped — so `Labels=a,b,c`, `Labels= a , b ,c`, and `Labels=[a, b, c]` all send the identical wire payload, and a lone scalar (`Labels=urgent`) becomes a one-element array. Element shaping and `allowedValues` validation are unchanged (labels stay bare strings, components/versions `{"name": …}`, options `{"value": …}`); sequence elements are never split, so `Labels=["a,b"]` remains the escape hatch for a literal comma inside one element, and `Labels=""` clears the field. Scalar (non-array) fields are untouched — a comma in `Summary=Fix bug, add test` stays literal. Because the split lives in the shared resolver, it covers CLI `jira write --set-field` / `jira create --set-field` and the MCP `jira_edit`/`jira_create` field maps alike; `--set-field` remains **replace**, not append. CLI `--dry-run` still prints the raw pre-conversion value (it deliberately makes no network call, so field schemas are unknown at that point). See [docs/user-guide.md](docs/user-guide.md#jira-reading-and-writing-issues).
- **Per-commit `in_main_branches` is now populated, and amend refuses already-pushed commits** ([#1105](https://github.com/rust-works/omni-dev/issues/1105)): two coupled gaps from the #1103 functionality-gap analysis are closed. First, the documented `commits[].in_main_branches` field ("array of remote main branches that contain this commit") had been a permanent empty stub since its introduction — every commit, including ones on `origin/main`, reported `[]`, so any consumer (human or AI) inferred every commit was unpushed. A new [`src/git/main_branches.rs`](src/git/main_branches.rs) module resolves each remote's main-branch tip **locally and offline** (symbolic `refs/remotes/<remote>/HEAD`, else the `main`/`master`/`develop` common-name fallback; remotes that resolve neither are skipped conservatively rather than guessed) and checks containment per commit via the git2 commit graph — tips are computed once per invocation, not per commit. Second, that containment now gates the amend path: `validate_commit_amendable` (the chokepoint all three amend surfaces — `amend`, `twiddle`, and `check --twiddle` — funnel through) refuses to amend a commit contained in a remote main branch, naming the short hash, the containing branches, and the escape hatch. A new `--allow-pushed` flag on `amend` and `twiddle` overrides the refusal (with a printed warning); the MCP `git_twiddle_commits` tool deliberately does **not** expose the override — an AI caller silently rewriting published history is exactly the failure mode the guard exists for, so overriding requires a human running the CLI flag. (`CommitInfo::from_git_commit` gains a `main_tips` parameter — technically API-breaking for library consumers.) See the [user guide](docs/user-guide.md#safety-already-pushed-commits).
- **MCP `jira_edit` tool — set arbitrary fields on an existing issue by name** ([#1157](https://github.com/rust-works/omni-dev/issues/1157)): once a ticket was created, changing labels or custom fields (Acceptance Criteria, Story Points, …) from MCP forced a fallback to raw `curl -X PUT /rest/api/3/issue/{key}` with a hand-authored payload — `jira_write`'s `fields` escape hatch takes only canonical ids with already-API-shaped values. The new `jira_edit` tool takes an issue key plus a `{field-name-or-id: value}` map, resolves names against the issue's edit screen (`editmeta`), and coerces each value to the API shape with the same shared resolver the CLI uses: select/option strings become `{"value": …}`, multi-selects arrays of those, labels plain string arrays, number/date scalars pass through, issue-link fields (e.g. `Parent`) wrap an issue key as `{"key": …}`, and rich-text fields accept JFM markdown (auto-converted to ADF; `""` clears) or a raw ADF document object (`{"type": "doc", …}`, validated and forwarded as-is). Supports `dry_run: true` (previews the exact `PUT` method/path/body like the other mutating tools) and returns `{status: ok, key, updated_fields}` as YAML. Unlike `jira_write`'s pass-through hatch, an `editmeta` failure is fatal — resolution is the tool's contract. Mirrors the CLI's existing `omni-dev atlassian jira write --set-field`; no new CLI subcommand (the CLI `jira edit` name belongs to the interactive editor). See [docs/mcp.md](docs/mcp.md).
- **`*_path` file alternatives to inline bodies on write-side MCP tools** ([#1093](https://github.com/rust-works/omni-dev/issues/1093)): an AI caller pays an O(size) generation cost to emit a large body inline through a tool call — the model writes every byte through its output stream — so writing a 70 KB Confluence page could take minutes just to *emit the tool call*, versus seconds for the same bytes read from disk. This is the write-side mirror of `confluence_read`/`jira_read`'s `output_file` (which already solved the symmetric read-side cost). Every text body a write tool accepts inline now has an optional sibling parameter the server reads from a filesystem path instead: `confluence_write`/`jira_write` gain `content_path`; `confluence_create` gains `content_path` + `document_path`; `jira_create` gains `description_path` + `document_path`; `confluence_comment_add`/`confluence_comment_add_inline` gain `content_path`; `jira_comment`/`jira_comment_edit` gain `body_path`; and `ai_chat` gains `message_path`. The inline and path forms are **mutually exclusive** (supplying both errors before any network call), and where the body is required exactly one must be given. A shared [`src/mcp/content_input.rs`](src/mcp/content_input.rs) helper (`resolve_content_input`/`require_content_input`) centralises the either/or resolution and mutual-exclusion check, and each tool's description now says to prefer the path form when the body is already on disk. The engine now owns a background tokio task (`SnowflakeEngine::start_heartbeat`, spawned by the daemon's `SnowflakeService`, cancelled and awaited on shutdown — the `BridgeServer` CancellationToken pattern) that every `SNOWFLAKE_HEARTBEAT_INTERVAL` (default 900s, ¼ of the 3600s session-token validity; `0` disables) visits each pool's **idle** sessions: each is borrowed through the existing permit + checkout machinery (so a query arriving mid-heartbeat briefly waits instead of authenticating a duplicate session), proactively renewed if its session token would lapse before the next tick, heartbeated, and restored without touching `last_used`/context bookkeeping (a keep-alive is not a query). The settings write side now mirrors the read side's profile targeting: new `Settings::upsert_env_vars_in` / `Settings::remove_env_vars_in` take an explicit `Option<&str>` profile and write to `profiles.<name>.env` when one is given (the existing base-`env` helpers delegate with `None`, so external callers are unaffected), and all three auth flows resolve the active profile (`--profile` / `OMNI_DEV_PROFILE`) at the production boundary and thread it down as a plain parameter (issue #1030 style). A profile-targeted logout removes only that profile's keys, leaving the base map and other profiles intact, and the login/logout confirmation messages name the profile they wrote to (`… settings.json (profile 'work')`). See the new [Populating a profile](docs/configuration-best-practices.md#populating-a-profile) section.
- **`confluence attachment delete` gains `--dry-run` and joins the shared destructive-command guard** ([#1126](https://github.com/rust-works/omni-dev/issues/1126)): the attachment-delete CLI command carried `--force` + `--purge` but not `--dry-run`, while every sibling removal (`confluence delete`, `confluence label remove`, `jira link remove`, `jira watcher remove`) offered all of them — so the ADR-0027 trick of sanity-checking a scripted `--force` invocation by appending `--dry-run` silently didn't exist for attachments. The command now accepts `--dry-run` (prints `Would delete attachment <ID>.` — or `Would permanently purge attachment <ID>.` with `--purge` — and exits without any API call; takes precedence over `--force`) and is migrated off its private `confirm_with_reader` copy onto the shared `guard_destructive_with_io` helper in [`src/cli/atlassian/confirm.rs`](src/cli/atlassian/confirm.rs), the third duplicated prompt copy ADR-0027 set out to eliminate. Like label removal, the preview echoes the ID without a lookup, keeping the no-API-call contract strict. Prompt wording, `--force`, and `--purge` behaviour are unchanged; the `confluence_attachment_delete` MCP tool intentionally stays confirm-free/`--force`-shaped per ADR-0027's MCP convention. The broader #1126 question — a uniform `--dry-run`/`--force` rule across *all* mutating CLI commands — remains open. See [ADR-0027](docs/adrs/adr-0027.md) and the destructive-commands section of [docs/user-guide.md](docs/user-guide.md).
- **Custom-field resolver: labels, issue-link (`Parent`), raw-ADF passthrough, and system-field ids** ([#1157](https://github.com/rust-works/omni-dev/issues/1157)): four gaps in the shared field resolver behind `jira create --set-field` / `jira write --set-field` / MCP `jira_create.custom_fields` (and the new `jira_edit`) are closed. **Labels**: array fields were always coerced to option objects (`[{"value": …}]`), which the API rejects for `labels` and labels-type custom fields — the array arm now dispatches on the schema's `items` type (`string` → plain string array, `component`/`version` → `[{"name": …}]`, options unchanged), so `Labels=[a, b]` finally works. **Issue-link fields**: schema type `issuelink` (e.g. `Parent`) was rejected outright ("Unsupported field type 'issuelink'"), forcing a create-then-`jira_link_parent` two-step when bulk-creating under one parent — an issue key string now coerces to `{"key": …}` at create and edit time. **Raw ADF**: rich-text fields only accepted JFM strings; a mapping with `type: doc` is now validated against the ADF content model and forwarded verbatim, the escape hatch for structured content JFM can't express. **System ids**: field lookup only honoured ids matching `customfield_<digits>`, so canonical system ids like `labels`/`parent`/`priority` never resolved — the id lookup is now unconditional (an exact id match shadows an identically spelled display name). Under the hood, `EditMetaSchema` now captures the `items` and `system` properties from editmeta/createmeta, and `is_adf_rich_text` also recognises the system `description`/`environment` fields, so a string sent to those via any `fields` map converts JFM→ADF instead of drawing a 400.
- **Credential structs no longer derive a plaintext `Debug` (latent secret leak)** ([#1131](https://github.com/rust-works/omni-dev/issues/1131)): `AtlassianCredentials.api_token`, `DatadogCredentials.api_key`/`app_key` (and `DatadogClient`'s stored copies), and the Snowflake session/master tokens all derived a `Debug` that printed the secret verbatim, so any future `tracing::debug!("{x:?}")` or `.context(format!("… {x:?}"))` would have leaked it into logs or error chains. Those fields are now wrapped in a new redacting newtype, [`utils::secret::Secret`](src/utils/secret.rs), whose hand-written `Debug` prints `<redacted>` and which implements neither `Display` nor serde — the value can only leave the wrapper through an explicit, greppable `expose_secret()` call at the HTTP-header/settings-write boundaries. The Debug-formatting tests, which previously only asserted the type *name* appeared (and so would have passed even while leaking), now assert the secret value is absent and `<redacted>` is present. Found in the #1101 security review. (Field type change on the two `pub` credential structs is technically API-breaking for library consumers.)
- **Daemon runtime files are created with tight permissions atomically — no write-then-chmod window** ([#1132](https://github.com/rust-works/omni-dev/issues/1132), [#1139](https://github.com/rust-works/omni-dev/issues/1139)): two low-severity findings from the [#1101](https://github.com/rust-works/omni-dev/issues/1101) security review shared one root pattern — creating a path under the ambient umask and only *then* tightening it, leaving a brief window where the inode was group/world-readable. The bridge token file was written with `std::fs::write` then `chmod`'d `0600` (#1132), and `ensure_dir_0700` did `create_dir_all` then `chmod 0700` (#1139). Both now pass the mode to the creating syscall itself: directories are born `0700` via `DirBuilder::mode`, and a new `paths::write_file_0600` opens the token file `O_CREAT` with mode `0600` (matching the `0600`-from-birth posture the daemon socket has had since [#995](https://github.com/rust-works/omni-dev/issues/995)). Additionally, modes set only at creation are now re-applied to pre-existing looser-perm files: `write_token` re-tightens a stale loose `bridge.token` *before* the new secret lands in it, and the request log re-tightens a loose `log.jsonl` (an older version's, or a user-set `OMNI_DEV_LOG_FILE` target) on every open — both via handle-based `fchmod` (`paths::ensure_handle_0600`), so there is no path race and no umask interference. Defense-in-depth only: the parent directories were already `0700`, so no privilege boundary was crossable in practice.
- **JFM→ADF validation failures now point at the source line, column, and offending text** ([#1087](https://github.com/rust-works/omni-dev/issues/1087)): when a JFM body converts to ADF that violates the content model — most commonly bold-wrapping-inline-code (`` **`/api/v1/example`** ``, which maps to a text node carrying both `strong` and `code`, a combination ADF forbids) — the write was aborted with only an opaque ADF index path (e.g. `at /38/4/0/1`), leaving the author to count siblings under an unrendered tree to find the culprit in a large document. The [`AdfValidationError`](src/atlassian/adf_validated.rs) is now enriched: it resolves the index path against the produced ADF tree to name the **offending run's text** (`in text: "/api/v1/example"`) and, when the JFM source is available, reports the **1-based `line:column`** of that run in the original markdown (columns counted in characters, so multi-byte text lines up with an editor cursor). Every validation error gains the text excerpt via [`ValidatedAdfDocument::try_new`](src/atlassian/adf_validated.rs); the JFM-sourced paths additionally gain `line:column` through a new shared [`markdown_to_validated_adf`](src/atlassian/adf_validated.rs) helper (and `try_new_with_source` / `validate_with_source` variants) now used by the MCP `jira`/`confluence` create/write/comment tools, the matching CLI commands and their `--dry-run` / `convert to-adf` preflights, and custom-field rich-text sections. This is the diagnostics improvement from #1087 option 2; silently splitting the `strong`+`code` marks (option 1) remains a separate follow-up.

- **`claude-cli` budget cap no longer fails open silently on an invalid value** ([#1135](https://github.com/rust-works/omni-dev/issues/1135)): the `--claude-cli-max-budget-usd` / `OMNI_DEV_CLAUDE_CLI_MAX_BUDGET_USD` spending cap dropped non-positive, non-finite, or unparseable values without a trace — a typo like `--claude-cli-max-budget-usd 0` ran the invocation with **no cap at all** and no signal that the guardrail was gone. A set-but-unusable value now logs a `WARN` naming the offending value and the env var before running uncapped (an unset cap stays silent, as before). The behaviour is still fail-open by design — see [ADR-0028](docs/adrs/adr-0028.md). [docs/ai-backends.md](docs/ai-backends.md#spending-cap) now also documents the cap's two inherent limitations from the #1101 security review: enforcement is delegated to `claude -p` (omni-dev has no independent kill switch), and because a turn's cost is only known after it completes, a single invocation can overshoot the cap by up to one model turn.
- **launchd socket activation no longer leaks extra file descriptors** ([#1145](https://github.com/rust-works/omni-dev/issues/1145)): when launchd hands over the activated socket, the kernel can supply more descriptors than the single one declared in the plist; the daemon adopted the first as its Tokio listener and silently left any extras open for the process lifetime (an fd leak). `daemon run` now detects `cnt > 1`, logs a warning, and closes each extra descriptor. macOS-only, on the launchd socket-activation path.
- **`estimate_lines_changed` no longer subtracts on a malformed diff-stat token** ([#1151](https://github.com/rust-works/omni-dev/issues/1151)): the diff-stat estimator parsed the count after `|` as `i32` and summed it, so a malformed line with a negative token (e.g. `|-1`, reachable because a vertical tab counts as `split_whitespace` whitespace) drove the running total negative — the case a property test shrank to. `git diff --stat` counts are always non-negative, so the token is now parsed as `u32`: a stray negative or non-numeric token fails to parse and is skipped instead of subtracting. A pre-existing latent bug surfaced during the #1151 refactor; two deterministic tests pin the previously proptest-only fall-through branches.

### Security
- **`claude-cli` tool-access escape hatch scrubs well-known secret env vars; untrusted-input caveat documented** ([#1144](https://github.com/rust-works/omni-dev/issues/1144)): enabling `--claude-cli-allow-tools` removes the two mitigations that normally contain the `claude-cli` backend — tools-off and prompt-is-just-data — while the nested `claude -p` session inherits the parent environment and is prompted with attacker-influenceable content (diffs, commit messages, JIRA text), so a prompt-injection payload could in principle drive Bash/Read tool calls that exfiltrate inherited credentials. As a compensating control, the tool-access hatch now widens the existing env scrub (`CLAUDE_PROJECT_DIR`, `CLAUDE_CODE_*`, `CLAUDE_PROJECT_*`) to well-known secret vars: suffixes `_API_KEY`/`_TOKEN`/`_SECRET`/`_PASSWORD`/`_CREDENTIALS` plus `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`. `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` are always kept (the nested `claude` may authenticate through them; omni-dev passes it no credentials of its own), and a new `OMNI_DEV_CLAUDE_CLI_KEEP_ENV` (comma-separated exact names) exempts credentials a tool-enabled workflow legitimately needs, e.g. `GITHUB_TOKEN` for env-authenticated `gh`. The scrub is inactive on the default (tools-off) path, secret-var removals are logged at `DEBUG` (names only), and the per-invocation `WARN` now spells out the untrusted-input risk — which is also documented next to the escape hatch in [docs/ai-backends.md](docs/ai-backends.md#escape-hatch-tool-access) and [ADR-0028](docs/adrs/adr-0028.md). Info severity, from the #1101 security review.
- **JFM→ADF conversion now bounds nesting depth instead of overflowing the stack** ([#1130](https://github.com/rust-works/omni-dev/issues/1130)): the hand-written markdown parser behind `to-adf` recursed once per nesting level with no limit, and a Rust stack overflow is an uncatchable abort (guard page, not a panic) — so a few KB of malicious input (`> > > …` at ~2 bytes per level, or equivalently nested lists, `:::panel`/`:::expand` directives, `:::td` table cells, or layout columns) killed the whole process. Reachable from the MCP `atlassian_convert` tool with fully caller-supplied content (a prompt-injection scenario that takes down the long-lived MCP server/daemon), `omni-dev atlassian convert to-adf`, and every JFM write path. A `depth` counter is now threaded through every nested `MarkdownParser` and checked in `parse_blocks` against a 64-level cap, returning a clean "maximum depth" error through the existing fallible pipeline (the cap is below serde_json's 128 — which already bounds the opposite `from-adf` direction — so that at-cap parsing also fits debug-build stacks; real documents nest under 20 levels). The independent **inline** recursion vector (nested emphasis/strikethrough/link text/bracketed spans/inline `:span` directives, ~2–3 bytes per level) is capped at the same depth but degrades gracefully — past the cap the remaining run is emitted as literal text instead of erroring, since inline parsing is infallible by design. Found during the #1101 security review.
- **`worktrees list` strips terminal control bytes from registry strings** ([#1137](https://github.com/rust-works/omni-dev/issues/1137)): a same-user process that can write the daemon socket can `register` a worktrees entry whose `title`/`repo`/`folders` contain ANSI/terminal control sequences, and the `omni-dev worktrees list` table printed those fields verbatim — letting a registered entry move the cursor, clear the screen, retitle the terminal, or forge extra table rows on the operator's TTY. The table renderer now strips control characters (C0 including ESC/CR/LF/BEL, DEL, and C1) from `repo`, `title`, and folder strings before formatting. Display-layer only: the wire payload and `worktrees list --json` stay byte-verbatim per the companion contract (JSON encoding already escapes control bytes), and native tray menus never interpreted ANSI. Found during the [#1101](https://github.com/rust-works/omni-dev/issues/1101) security review; see [docs/worktrees-service.md](docs/worktrees-service.md#security).
- **Argv is scrubbed before it reaches the request log** ([#1129](https://github.com/rust-works/omni-dev/issues/1129)): the invocation record wrote the full process argv to `log.jsonl` verbatim — unredacted and not gated by `OMNI_DEV_LOG_BODIES`/`OMNI_DEV_LOG_HEADERS` — so a secret passed on the command line (e.g. `browser bridge request --header "Authorization: Bearer <secret>"` or `--body "<secret>"`) persisted in plaintext, in telling contrast to the `OMNI_DEV_*` env snapshot on the same record, which was already redacted. `command_line` is now scrubbed centrally in `record_invocation` before writing, in both `--flag value` and `--flag=value` forms: `--header` values naming a sensitive header (the same list `redact_headers` uses) keep the name but lose the value (`Authorization: REDACTED`), inline `--body` values are redacted (`@file` references are kept — a path, not a secret), and the value of any flag whose name contains a `token`/`secret`/`password`/`passwd`/`key` segment is redacted as future-proofing (flags ending in `-file`/`-path` carry paths and are exempt, e.g. `--token-file`). Write-side scrubbing means the on-disk line is clean, so `--grep`/`--query` matching and every output format are covered with no reader changes. Found during the #1101 security review; see the redaction posture in [docs/log.md](docs/log.md#redaction-posture).
- **Attachment downloads sanitize server-controlled filenames (path traversal / arbitrary file write)** ([#1127](https://github.com/rust-works/omni-dev/issues/1127)): the single-attachment download paths joined the remote-supplied filename/title onto the output directory verbatim, and `Path::join` both replaces the base on an absolute component and leaves `..` segments intact — so an attachment named `/Users/<victim>/.zshrc` or `../../../../.ssh/authorized_keys` on a page/issue the victim downloads would write attacker-controlled bytes to an attacker-chosen path, even in the default temp-dir configuration. Four surfaces were affected: the MCP tools `confluence_attachment_download`, `jira_attachment_download`, and `jira_attachment_images`, plus the CLI commands `jira attachment download|images` and `confluence attachment download` (the bulk `confluence download` command was already safe). All now route the remote name through the same sanitizer that command used — [`utils::path::attachment_filename`](src/utils/path.rs) reduces it to its final path component and falls back to `attachment-<id>` when no usable name remains — so the file always lands inside the intended directory. Explicitly user-supplied output *file* paths are untouched; YAML/console metadata still reports the original remote title, while `path` reports the sanitized on-disk location. Found during the #1101 security review.
- **Request log no longer records secret-bearing URL query/fragment values** ([#1133](https://github.com/rust-works/omni-dev/issues/1133)): the request log wrote the request URL verbatim, so a browser-bridge request to an operator-supplied target carrying credentials in the URL — a presigned S3/GCS URL, `?access_token=…`, an Azure SAS `?sig=…`, or an OAuth implicit-flow `#access_token=…` fragment — landed in `log.jsonl` unredacted, violating the log's "no secret material is ever written" contract (first-party clients were unaffected: Datadog/Atlassian/Snowflake/Claude authenticate via headers, which were already redacted). [`record_http_with`](src/request_log.rs) now centrally redacts, always-on, covering every current and future caller: query and fragment parameter values whose keys look secret-bearing (suffixes `token`/`secret`/`password`/`passwd`/`signature`/`api_key`; exact keys `sig`/`sas`/`jwt`/`auth`; prefixes `X-Amz-*`/`X-Goog-*`) are replaced with `REDACTED`, matching segments only — host, path, parameter keys, and all benign segments are preserved byte-verbatim, so `omni-dev log --url <SUBSTR>` filtering keeps working and clean URLs round-trip unchanged. Found in the [#1101](https://github.com/rust-works/omni-dev/issues/1101) security review; see the updated redaction posture in [docs/log.md](docs/log.md#redaction-posture).
- **Invocation-record argv also redacts secret-bearing URL query/fragment values** ([#1162](https://github.com/rust-works/omni-dev/issues/1162)): closing the #1133 URL leak on `kind:"http"` records surfaced the same secret reaching `log.jsonl` through a second field — the `kind:"invocation"` record's full argv. The #1129 argv scrub redacts secret-bearing *flag names* and `--header`/`--body` values, but a URL passed as `--url /api/export?access_token=…&sig=…` slipped through: `--url` is not a secret-bearing flag name, and its value is a whole URL, not a bare secret. `scrub_argv` now runs every resulting element through the same `redact_url` machinery `record_http_with` uses (a second layer after the flag-value scrub), so a credential carried in a URL argument's query or fragment — most naturally the browser bridge's operator-supplied target, or a presigned URL — is redacted (`access_token=REDACTED&sig=REDACTED`) while benign argv (paths, ports, non-secret params) passes through byte-identical, keeping `--grep`/`--query` matching intact. Found during end-to-end verification of the #1133 fix; see the redaction posture in [docs/log.md](docs/log.md#redaction-posture).
- **Worktrees registry capped at 256 entries** ([#1140](https://github.com/rust-works/omni-dev/issues/1140)): the daemon's worktrees service inserted `register` requests into an unbounded in-memory map, so a misbehaving same-user process flooding `register` with distinct keys could grow daemon memory until the 30 s TTL reaped the entries (TTL × request rate). The registry now holds at most 256 windows: where the TTL bounds how *stale* an entry can get, the cap bounds how *many* can exist. A `register` of a new key at the cap evicts the longest-silent entry (oldest `last_seen`, ties broken by key) instead of rejecting — `register` stays infallible so the companion-extension contract gains no error path, and an evicted live window self-heals through the existing `heartbeat → {known: false} → re-register` resync; upserts of an existing key never evict. Robustness hardening within the same-user trust domain (Info severity, from the #1101 security review) — no trust boundary changes. See [ADR-0040](docs/adrs/adr-0040.md) and [docs/worktrees-service.md](docs/worktrees-service.md).
- **settings.json credential store is no longer world-readable** ([#1128](https://github.com/rust-works/omni-dev/issues/1128), from the #1101 security review): `auth login` wrote the plaintext `ATLASSIAN_API_TOKEN` / Datadog keys (and, since the profile system shares the file, every profile's secret `env` bundle) to `~/.omni-dev/settings.json` with umask permissions — `0644` inside a `0755` directory on a typical host, readable by any local user on a shared machine. Every settings write now goes through the single hardened write site in [`src/utils/settings.rs`](src/utils/settings.rs): the parent directory is created/tightened to `0700`, the file is created through a `0600` handle (no window where a fresh file is world-readable), and a pre-existing looser-permission file is re-tightened to `0600` on every write — the same owner-only posture as the daemon's runtime state (`src/daemon/paths.rs`). Run any `omni-dev atlassian auth login` / `omni-dev datadog auth login` once to tighten an existing file, or `chmod 600 ~/.omni-dev/settings.json && chmod 700 ~/.omni-dev` by hand.
- **Request log redacts secret-bearing headers by name substring, not just an exact list** ([#1134](https://github.com/rust-works/omni-dev/issues/1134)): header redaction was an exact-name denylist, so an off-list auth header (`x-auth-token`, `x-amz-security-token`, `x-goog-api-key`, a custom `x-<vendor>-token`) would be written in the clear once a caller starts populating `HttpExtra` headers. A `SENSITIVE_HEADER_MARKERS` substring denylist — mirroring the `SECRETISH` env-var convention — is added alongside the exact list, so any header whose lowercased name is on the list or contains a marker is redacted (false positives redact harmlessly). Found in the #1101 security review; see [docs/log.md](docs/log.md#redaction-posture).
- **Facebook harvester no longer panics on a multi-byte char boundary** ([#1136](https://github.com/rust-works/omni-dev/issues/1136)): the fixed-size search windows in `parse_session_from_me` (anchor + 4000) and `refetch_doc_in_bundle` (at + 2000) sliced at an arbitrary byte offset that could land inside a multi-byte UTF-8 sequence, panicking with "byte index is not a char boundary" instead of returning the staged error — reachable because timeline HTML embeds other users' Unicode (emoji, non-ASCII names), i.e. semi-trusted input. A `search_window` helper now walks the end back to the nearest char boundary (shrinking the window by at most 3 bytes), used at both sites; regression tests straddle a 4-byte scalar across each window edge. Found during the #1101 security review.

### Added
- **Worktrees daemon service + `omni-dev worktrees list`** ([#1011](https://github.com/rust-works/omni-dev/issues/1011)): the daemon's third service (after the browser bridge and Snowflake) tracks the repos and git worktrees open across **every** VS Code window — something no single extension can do, since the extension host is sandboxed per-window and each window sees only its own `workspace.workspaceFolders`. A resident daemon is the rendezvous point that aggregates per-window registrations into one consistent, liveness-aware view served back to the CLI and tray. `src/daemon/services/worktrees.rs` holds an in-memory registry behind a `std::Mutex` (never held across `.await`) supporting four ops — `register` (idempotent upsert), `heartbeat` (returns `{known}` so a window re-registers after a daemon restart), `unregister`, and `list`. Liveness is **reap-on-read**: entries carry `last_seen` and a 30s TTL (three missed ~10s heartbeats) ages out a window that crashed without a clean `unregister`, run inline on every `list`/`status`/`menu` with no background task. A "Worktrees" tray submenu lists open windows with a best-effort `focus:<key>` action per window that shells out `code <folder>` (resolved via `OMNI_DEV_VSCODE_BIN` → well-known paths → bare `code`). The new read-only client is `omni-dev worktrees list [--json]`; the register/heartbeat/unregister ops are spoken directly over the daemon's existing `0600` NDJSON socket by a companion VS Code extension (a separate deliverable). No new trust boundary and nothing is persisted. See [ADR-0040](docs/adrs/adr-0040.md) and [docs/worktrees-service.md](docs/worktrees-service.md).
- **MCP `jira_bulk_create` tool for epic decomposition** ([#1055](https://github.com/rust-works/omni-dev/issues/1055)): a new tool that creates multiple JIRA issues **and** wires the dependency links between them in a single call, eliminating the per-call round-trips of sequential creation. It takes a batch of issue specs (each with a local `alias`) and link specs whose endpoints are resolved **alias-first** (to the freshly-minted keys) then as literal keys for pre-existing issues — so a whole epic + stories + sub-task links land at once. Each issue create mirrors the `jira_create` JFM→ADF path; execution is **continue-on-error** with an optional `fail_fast` to stop at the first failure, and the tool returns a structured YAML report (per-issue `alias`/`ok`/`key`/`self_url` or `error`, per-link outcomes, and summary tallies). An empty issues array allows linking existing keys only. **No rollback** — partial success leaves the created issues in place; see [docs/mcp.md](docs/mcp.md).
- **`custom_fields` on the MCP `jira_create` tool** ([#1052](https://github.com/rust-works/omni-dev/issues/1052)): `jira_create` gains an optional `custom_fields` map (field name **or** canonical `customfield_NNNNN` id → value). When present, the tool fetches `createmeta` to resolve human-readable names to API ids and shapes each value to its schema (e.g. wrapping a select as `{"value": …}`), mirroring the CLI's `--set-field`. This lets MCP clients satisfy fields a JIRA project marks **required at create time**, preventing the HTTP 400 (`customfield_NNNNN is required`) rejection. The `createmeta` fetch is skipped when no custom fields are supplied (fast path).
- **Client-side option-value validation for `jira create --set-field`** ([#1053](https://github.com/rust-works/omni-dev/issues/1053)): supplied values for enumerated option fields (single/multi-select) are now validated against the field's `allowedValues` from the JIRA edit meta **before** the API call, so an out-of-range value is rejected locally with a field-named, actionable error instead of an opaque server 400. Fields that don't enumerate values (free text, numbers, user pickers, cascading selects) skip local validation and defer to the API. `jira create --help` now documents the full `--set-field "NAME=VALUE"` semantics: splitting on the first `=` only, VALUE kept verbatim, exact case-sensitive matching, `customfield_<digits>`-or-exact-display-name resolution, the create-screen requirement, and that `--set-field` overrides frontmatter `custom_fields:`.
- **MCP `jira_create` / `confluence_create` accept a full JFM `document`** ([#1058](https://github.com/rust-works/omni-dev/issues/1058)): `jira_read` / `confluence_read` emit a full JFM document (frontmatter + body), but the create tools previously took only structured params with no way to accept a document — so the natural *read → edit → create* round-trip an agent expects was broken at the MCP layer (the CLI side was handled by #1050/#1051). Both create tools gain an optional **`document`** parameter carrying the JFM document: `project`/`summary`/`issue_type`, labels and **custom fields** (JIRA, via the same `createmeta` resolution the CLI uses) or `space_key`/`title`/`parent_id` (Confluence) are taken from its frontmatter — the JIRA project derives from `key:` when no `project:` is set — and the body becomes the description/page body. Passing `document` together with `description`/`content` is an error. The frontmatter resolution and the createmeta-aware create path are extracted into a shared `src/atlassian/create.rs` used by **both** the CLI and MCP surfaces, so they cannot drift. See [docs/specs/jfm.md](docs/specs/jfm.md#creating-from-frontmatter).

Applies to both `--format jfm` and `--format adf`.
- **JIRA create-screen introspection (`jira project create-meta`)** ([#1057](https://github.com/rust-works/omni-dev/issues/1057)): a new CLI command `omni-dev atlassian jira project create-meta --project <P> --issue-type <T>` and matching MCP tool `jira_project_create_meta` that answer "what does it take to create a `<Issue Type>` in `<Project>`?" up front, instead of forcing the create→HTTP 400 (`customfield_NNNNN is required`)→`field list`→`field options` recovery loop. - **`dry_run` on mutating Atlassian MCP tools** ([#1048](https://github.com/rust-works/omni-dev/issues/1048)): the CLI has long offered `--dry-run` on `jira create`/`write` and `confluence create`/`write` to preview a mutation without contacting the API, but the MCP surface exposed no equivalent — an AI agent uncertain about required fields or JFM→ADF formatting had no safe way to validate a call before committing it, and a failed call could leave a real, un-undoable ticket. For `jira_link_remove`, a dry-run previews the `DELETE` without requiring the destructive `confirm` guard. Each tool's description now points to the flag as the first thing to try when unsure. - **`jira create --help` documents the JFM frontmatter fields; `instance:` clarified as informational** ([#1051](https://github.com/rust-works/omni-dev/issues/1051)): `jira create --help` gained an epilogue listing the fields read from JFM frontmatter (`project`, `summary`, `issue_type`, `labels`, `custom_fields`, and `instance`), so the flag/frontmatter split is discoverable from `--help` alone — closing the agent-facing discoverability gap that motivated the issue (an agent reading `--help` had no signal these existed, and hit a runtime parse error as the first hint). - **macOS daemon switched to launchd socket activation** ([#1081](https://github.com/rust-works/omni-dev/issues/1081)): the `RunAtLoad`/`KeepAlive` LaunchAgent is replaced by a socket-activated one — launchd now **owns** the control socket (a `Sockets`→`Listener` dict, created `0o600`) and demand-spawns `omni-dev daemon run` on the first client connect, handing it the listening file descriptor; `daemon run` adopts that fd via the `launch_activate_socket(3)` FFI (`launchd::launchd_listener`) and falls back to binding the socket itself when not launchd-activated (a manual `daemon run`, CI, or any non-macOS platform). This makes the on-demand-only-mode start failure **architecturally impossible** and retires the tactical `launchctl kickstart` workaround from [#1078](https://github.com/rust-works/omni-dev/issues/1078): there is nothing for `RunAtLoad` to fail to do because the daemon is no longer run-at-load. Knock-on wins: cold-start latency for the first CLI invocation after login drops to ~0 (launchd is already listening), and a crashed daemon is re-activated on the next connect for free (no `KeepAlive` race). Behavioural notes: `daemon start` is now effectively a one-time install (after it, any CLI call re-activates the daemon — and login re-bootstraps the agent); `daemon stop` still boots the agent out for a real, observable stop (the demand socket is removed, so it stays down until the next `start` or login); and `daemon status` reporting "not running" now means "no process is resident right now", not "unavailable", since any connect re-activates it. Users upgrading from an older `RunAtLoad` daemon should run `omni-dev daemon restart` once to pick up the new plist. This change introduces the crate's first (narrowly-scoped, macOS-only) `unsafe` block for the FFI. See [ADR-0039](docs/adrs/adr-0039.md) and [docs/browser-bridge.md](docs/browser-bridge.md).

### Fixed
- **`jira create` no longer demands frontmatter when CLI flags are supplied** ([#1050](https://github.com/rust-works/omni-dev/issues/1050)): creating an issue previously required JFM frontmatter with `type`, `instance`, and `summary` even when the equivalent values were already on the command line, so a first invocation hit a chain of sequential parse errors (`Document must start with '---'…` → `missing field instance` → `missing field summary`), each needing an edit-and-retry — and `instance` had to be typed into the frontmatter only to be thrown away, since the create path builds its client from auth config, not the document. An agent that started with the MCP `jira_create` tool (which takes a plain description with no frontmatter) and switched to the CLI hit this wall immediately. Frontmatter is now **optional**: a file with no `---` block is treated entirely as the issue body with all metadata taken from flags — `omni-dev atlassian jira create --project P --summary "…" --type Story body.md` just works, matching MCP `jira_create`. When frontmatter *is* present, any field it omits falls back to the matching flag (extending the existing `--project`/`--type`/`--summary` override behaviour to the whole resolution), and `instance` is no longer required at all. Resolution precedence — **flags → frontmatter → derived/default** — is documented at the top of `create --help` and in [docs/user-guide.md](docs/user-guide.md). The strict read/write/edit round-trip (`JfmDocument::parse`) is unchanged; leniency is localized to the create path via a new `split_frontmatter` helper and a create-only `JiraCreateFrontmatter`.
- **Forbidden ADF mark combinations are rejected pre-flight with an actionable message** ([#1047](https://github.com/rust-works/omni-dev/issues/1047)): a JFM body like `` **`text`** `` (bold + monospace) maps to a text node carrying both `strong` and `code` marks, which ADF forbids — but Atlassian only signalled it as an opaque `HTTP 400: {"errorMessages":["INVALID_INPUT"],"errors":{}}` with no field, path, or offending node, leaving callers (and AI agents) to bisect the body by hand. The ADF mark validator now models the upstream schema's two mutually-exclusive inline-mark groups (the `code_inline_node` group `{annotation, code, link}` vs the `formatted_text_inline_node` group `{annotation, backgroundColor, em, link, strike, strong, subsup, textColor, underline}`, transcribed from the pinned `@atlaskit/adf-schema`) and emits a new `ForbiddenMarkCombination` violation when a text node mixes marks across groups — most commonly `code` paired with any styling mark. Because every create/write path (`jira create`, `confluence create`, `confluence write`) and `--dry-run` already run this validator before sending, the bad combination is now caught locally with a `cannot be combined with` message naming the two conflicting marks plus a remediation hint, instead of the opaque API error. Separately, `jira create` now routes its non-success responses through the same error handler `jira update` uses, so a field that reports it requires ADF surfaces as the actionable `JiraAdfFieldRequired` rather than a bare `ApiRequestFailed`. (Note: the issue also listed `link`+`textColor` as forbidden, but the pinned upstream schema permits both in the formatted group, so that pairing is intentionally **not** flagged.)
- **Close a process-global env race in config discovery** ([#821](https://github.com/rust-works/omni-dev/issues/821)): the env-reading functions in `claude::context::discovery` are split into thin public boundary wrappers and pure inner seams that take the environment as an injected dependency (STYLE-0028), so the 18 tests formerly serialised by a module-local `ENV_MUTEX` now inject a `MapEnv` and temp dirs instead of mutating process env — the mutex is deleted and the module is lock-free and fully parallel. This removes the last writer of `XDG_CONFIG_HOME` that raced a *different* lock (`AUTH_ENV_MUTEX`) over the same process-global var (the cross-module `EACCES` flake seen on PR #1014); public signatures are unchanged thin wrappers, so callers and the CLI surface are untouched. No user-facing behaviour change.

### Added
- **Extensible daemon for long-lived services** ([#987](https://github.com/rust-works/omni-dev/issues/987), [#988](https://github.com/rust-works/omni-dev/issues/988)): A new `omni-dev daemon` command tree (`run`/`start`/`stop`/`restart`/`status`) that supervises long-lived **services** inside one process behind a private per-user Unix-domain control socket. Single-instance supervision falls out of the exclusive socket bind itself (with a `ping`-probe + stale-socket reclaim), JSON/NDLINE operations route to registered services via a `ServiceRegistry`, and `SIGTERM`/`SIGINT` trigger graceful shutdown; on macOS, `daemon start` installs a launchd LaunchAgent for auto-start at login. The **browser bridge is the first service migrated onto it**: `BridgeService` wraps the same `BridgeServer` planes verbatim — standalone `omni-dev browser bridge serve` keeps working unchanged — persists its session token to a `0600` file so the thin clients (`request`, `harvest`) transparently discover a daemon-hosted bridge, and the bridge now also accepts WebSocket connections on IPv6 (`::1`) loopback alongside IPv4. An optional macOS **menu-bar tray** (behind the off-by-default `menu-bar` feature) renders one submenu per service from `DaemonService::menu()`, refreshes ~1 Hz, and offers copy-snippet / copy-bridge-key / copy-request-command actions; enabling it inverts the runtime so tokio runs on background threads and the main thread hosts the macOS event loop (all other paths unchanged). `daemon status` prints a per-service table (`--json` for machines). New modules under `src/daemon/`; hosting the bridge does **not** change its trust boundary. See [ADR-0039](docs/adrs/adr-0039.md), [ADR-0036](docs/adrs/adr-0036.md), and [docs/browser-bridge.md](docs/browser-bridge.md).
- **Graceful connection draining on daemon shutdown** ([#992](https://github.com/rust-works/omni-dev/issues/992)): accepted control-socket connections that have begun dispatch now always finish their reply before the daemon tears down, instead of being abandoned mid-request. Handlers are tracked in a `JoinSet` (reaped during normal operation so a long-lived daemon's set can't grow unbounded) and drained on shutdown under a 5-second `DRAIN_TIMEOUT`.
- **Local invocation + HTTP request log (`omni-dev log`)** ([#1025](https://github.com/rust-works/omni-dev/issues/1025)): A local, append-only NDJSON log (`log.jsonl`) records exactly one `kind:"invocation"` line per `omni-dev` run — the argv-derived subcommand path, exit code, duration, the rendered error chain on failure, and a whitelisted `OMNI_DEV_*` env snapshot — plus one `kind:"http"` line per outbound request, hooked **inside** each client's existing retry loop (Atlassian, Datadog, Snowflake, the direct-Anthropic/Bedrock/OpenAI-compatible Claude backends and the `claude-cli` subprocess, and the browser bridge) so retries and transport errors are captured too. HTTP records inherit their parent run's `invocation_id`/`source`/`mcp_tool` through a process-global context plus a `tokio` task-local override (the MCP server scopes `source=mcp` + the tool name around each `call_tool`), so every request correlates to the run that issued it. The new read-only `omni-dev log` subcommand searches and pretty-prints it: a filter matrix (`--since`/`--method`/`--status`/`--service`/`--command`/`--url`/`--grep`/`--fuzzy`/`--id`), a `--query` mini-language (AND/OR/NOT, parentheses, `field:value` terms, bare fuzzy tokens, status-class matching like `5xx`), `oneline`/`json`/`full` renderers (`json` emits the on-disk line verbatim so it composes with `jq`), and `-n/--limit` + `-f/--follow` streaming. Best-effort and default-on — a write failure never changes the caller's exit code — and **no secret is ever written**: auth headers are redacted centrally, request/response bodies are opt-in via `OMNI_DEV_LOG_BODIES` (headers via `OMNI_DEV_LOG_HEADERS`), and `OMNI_DEV_LOG_DISABLE` is an absolute opt-out. The path resolves via `OMNI_DEV_LOG_FILE` → `state_dir`/`data_dir` `omni-dev/log.jsonl` (dir `0700`, file `0600`). New module `src/request_log.rs` plus `src/cli/log.rs`; see [docs/log.md](docs/log.md).
- **Confluence attachment download** ([#1036](https://github.com/rust-works/omni-dev/issues/1036)): The Confluence side gains the attachment-download capability the JIRA side has had since #309, closing an asymmetry — previously the only ways to pull an attachment binary off a page were dropping out to `curl` with the `api_token` read from `~/.atlassian/config.yaml` (leaking credentials into shell pipelines the wrapper exists to avoid) or `confluence download`, which fetches page text but skips attachment binaries entirely. New CLI `omni-dev atlassian confluence attachment download <ATTACHMENT_ID> [--output <PATH>]` resolves the attachment's `downloadLink` from its ID (v2 `GET /wiki/api/v2/attachments/{id}`) and streams the binary to disk via the same authenticated, request-logged transport as the rest of the client (following the media-CDN redirect); `--output` defaults to the attachment's filename in the cwd, treats an existing directory as a target dir, and otherwise writes verbatim. New MCP tool `confluence_attachment_download` (`{attachment_id, output_path?}`) mirrors it, returning YAML `{id, title, media_type, bytes, path}` and writing to a fresh temp dir when `output_path` is omitted so the assistant can read the file back. Finally, `confluence download` (and the `confluence_download` MCP tool) gain `--include-attachments` / `include_attachments`, which fans out per page after the content write to fetch every attachment binary into an `attachments/` subdirectory beside its content file — making full-page snapshots possible in one command. The fan-out is best-effort (listing or per-binary failures are logged and swallowed so a broken attachment can't fail an otherwise-successful page) and strips remote-controlled attachment titles to a single path component to prevent directory traversal. New `ConfluenceApi::{get_attachment, download_attachment, download_attachment_bytes}`; see [docs/user-guide.md](docs/user-guide.md#confluence-attachments).
- **Per-video metadata sidecars for `transcript youtube sync`** ([#976](https://github.com/rust-works/omni-dev/issues/976)): `sync` now writes a metadata sidecar `<out>/<channel-id>/<video-id>.meta.yaml` alongside each transcript — title, channel, channel URL, category, `published_at`, duration, description, keywords, point-in-time `view_count`/`like_count`, `is_live_content`/`is_unlisted`, thumbnail, a `fetched_at` snapshot timestamp, and `schema: 1`. The metadata it already received per video was previously parsed and discarded. Sidecars are language-independent (one per video) and written atomically (temp file + rename). They are planned by a **filesystem scan** of the channel directory rather than channel re-enumeration, so **missing sidecars are backfilled on every run** for the full set of already-synced videos, without `--full` and without touching the bot-gated transcript path. The new **`--refresh-metadata-older-than <date-spec>`** flag re-fetches sidecars whose `fetched_at` predates the cutoff — `<date-spec>` accepts an absolute date (`YYYY-MM-DD`, midnight UTC), a full RFC 3339 timestamp, or a relative spec like `"2 days ago"` (units: minute, hour, day, week, month, year). Metadata is fetched with a single **un-gated WEB-client `/player` call** (no `visitorData` bootstrap), which carries the `microformat` block — publish date, like count, category — that the `ANDROID_VR` transcript path lacks; that independence is what makes standalone backfill/refresh possible. `microformat`-sourced fields are omitted when absent (ratings disabled, or private/removed videos written from `videoDetails` alone). Metadata fetch/write failures are tallied separately (`metadata synced`/`refreshed`/`failed` in the `SyncReport`) and **never block or fail transcript syncing**. New module `src/transcript/sources/youtube/metadata.rs`; see [docs/transcript.md](docs/transcript.md).
- **Snowflake query service on the daemon** ([#999](https://github.com/rust-works/omni-dev/issues/999)): A second daemon-hosted service (after the browser bridge) that authenticates Snowflake sessions via external-browser SSO and reuses them for **concurrent arbitrary SQL** across **any** account — solving the "SSO popup on every query" problem (a live Snowflake session has no serialization API, so a resident process is the only way to reuse one auth across invocations). New engine `src/snowflake/`: each `(account, user)` keeps a **bounded session pool** (`SNOWFLAKE_POOL_SIZE`, default 4) so queries run **concurrently** on one authentication identity while still honoring per-query `warehouse`/`role`/`database`/`schema` — applied as a `USE`-diff on checkout against each session's captured base context, with a `tokio::Semaphore` capping concurrency *and* the live-session/auth count, a shared auth gate serializing browser popups, lazy re-auth on session expiry, and arbitrary-schema → self-describing `{ columns, rows }` JSON conversion. Plus a thin `DaemonService` adapter `src/daemon/services/snowflake.rs` (routes `query`/`sessions`/`disconnect`, contributes a "Snowflake" tray submenu, surfaces in `daemon status`) and a thin client `omni-dev snowflake {query,sessions,disconnect}` — `query` takes SQL from an argument or stdin and supports per-query `--warehouse/--role/--database/--schema` plus `--format json|yaml`. Account/user/context default from `SNOWFLAKE_*` env vars then `~/.omni-dev/settings.json` (the Atlassian pattern); **no hardcoded accounts**. No new socket or persisted secret — it rides the daemon's existing `0600` Unix control socket and keeps sessions in memory only. The engine runs on an in-tree **clean-room Snowflake v1 REST client** (`src/snowflake/client/`, no third-party connector) that adds **session-token renewal** (so a session survives the ~1h token lifetime instead of re-authenticating), transparent renew-and-retry on expiry, **async-result polling** so heavy long-running queries complete (bounded by `SNOWFLAKE_QUERY_TIMEOUT`) instead of being killed by a per-request timeout, per-request HTTP timeouts (`SNOWFLAKE_HTTP_TIMEOUT`) and transient-error retries, an overall sign-in deadline (`SNOWFLAKE_AUTH_TIMEOUT`) so a hung auth can't block others, and external-chunk download for large results. No third-party Snowflake connector is used (an early `snowflake-connector-rs` dependency, which pulled `rsa`/RUSTSEC-2023-0071, was removed). See [docs/snowflake-service.md](docs/snowflake-service.md).

### Fixed
- **omni-dev compiles on Windows again — the daemon is gated to Unix** ([#1041](https://github.com/rust-works/omni-dev/issues/1041)): the new daemon (#987, #988) is built on Unix-domain sockets (`tokio::net::UnixListener`/`UnixStream`), which don't exist on Windows, so `cargo build` and `cargo install omni-dev` failed to compile on Windows MSVC (`error[E0432]: unresolved import` in `src/daemon/{server,client,single_instance}.rs`). The daemon runtime and its `daemon` / `snowflake` CLI subcommands are now gated behind `#[cfg(unix)]` and compiled out on Windows — running the daemon there remains future work — while `daemon::paths` stays cross-platform (the request log and the browser thin-client token discovery depend on it), so the rest of omni-dev builds and installs on Windows as before. A Windows build job was added to CI so this class of portability regression is caught on every PR rather than only when a release tag is pushed.
- **Editor-sized images with pixel widths above 100 round-trip without ADF validation failure** ([#1037](https://github.com/rust-works/omni-dev/issues/1037)): real Confluence emits `mediaSingle` pixel widths well above 100 for editor-sized images, but the pinned ADF schema validated `width` against a fixed `[0, 100]` range (the percentage cap), so a read→write round-trip of such an image failed validation. The validator now models the upstream schema's sibling-conditional range via a new `CondNumRange` attribute type resolved at validation time, before shape-checking: pixel widths (`widthType="pixel"`) are unbounded `[0, ∞)`, while percentage widths (the default when `widthType` is absent) keep the `[0, 100]` cap.
- **Git command spawning no longer races concurrent environment mutation** ([#1022](https://github.com/rust-works/omni-dev/issues/1022)): git subprocesses are now spawned through a `git_command()` helper that snapshots the environment (`env_clear()` + `envs(std::env::vars_os())`) under `std`'s lock before `exec`, eliminating intermittent failures where a concurrent `std::env::set_var` raced a child process's environment capture. The helper also sets `commit.gpgsign=false` on spawned git to avoid flaking on hosts with commit signing enabled.
- **Daemon control socket is owner-private from birth** ([#995](https://github.com/rust-works/omni-dev/issues/995)): the socket was created and then `chmod`-ed to `0600`, leaving a brief window where it could be group/world-accessible — security-critical for a socket carrying privileged daemon operations. Binding now tightens the process umask to `0o177` for the synchronous bind only (restored via an RAII guard), so the socket inode is never created with broader permissions.
- **Bridge session token is masked in the menu-bar tray** ([#997](https://github.com/rust-works/omni-dev/issues/997)): `BridgeService::menu` rendered the full session token as a visible `Key: <token>` label — needless exposure to shoulder-surfing, screen-sharing, and screenshots when the value is already reachable via the "Copy bridge key" action. The label now reveals only a masked tail (`Key: ••••<last 4>`, or fully masked when the token is too short to keep most of its entropy hidden); the full value stays behind the copy action.
- **`launchctl bootout` spawn failures are surfaced on `stop`/`restart`** ([#996](https://github.com/rust-works/omni-dev/issues/996)): the pre-bootstrap bootout and `unload`'s bootout both discarded spawn errors *and* non-zero status via `let _ = …output()`, so if `launchctl` was missing or the GUI domain unreachable the teardown silently did nothing yet `unload` returned `Ok` — making `stop`/`restart` claim to have disabled auto-start when they may not have. Both call sites now share a `bootout` helper that logs a spawn failure via `tracing::warn!` while still treating "already not loaded" as the benign no-op it should be.
- **Duplicate daemon service registrations are rejected** ([#998](https://github.com/rust-works/omni-dev/issues/998)): `ServiceRegistry::register` pushed unconditionally, so a second service sharing a `name()` was dead code for routing (`get`/`dispatch` use `.find`) yet still double-counted in `status`/`menu`/`shutdown` iteration. A duplicate is now dropped with a warning, keeping the documented first-wins contract consistent everywhere.
- **`daemon restart` no longer fails with "Bootstrap failed: 5: Input/output error"** ([#1004](https://github.com/rust-works/omni-dev/issues/1004)): `launchctl bootout` is asynchronous and returns before launchd has finished tearing the job down, so an immediate re-bootstrap raced the in-flight teardown into EIO (the socket-ping `wait_until_down` never waited for the launchd job itself to unload). `install_and_load` now polls `launchctl print` until the job leaves the domain before bootstrapping, and retries bootstrap a few times to absorb any residual transient EIO.
- **`daemon start` no longer times out when launchd downgrades `RunAtLoad`** ([#1078](https://github.com/rust-works/omni-dev/issues/1078)): `launchctl bootstrap` returned success but launchd intermittently downgraded the plist's `RunAtLoad` to on-demand activation, leaving the freshly-bootstrapped job parked at `state = not running` (`pended nondemand spawn = speculative`) indefinitely — so `start`/`restart` burned their 5s readiness budget and errored with `daemon did not become ready within 5s`, even though the plist was valid and `daemon run` worked in the foreground. The downgrade is non-deterministic (depends on launchd's internal scheduling, not the plist), so depending on `RunAtLoad` to spawn the process was inherently racy. `install_and_load` now issues an explicit `launchctl kickstart -p <domain>/<label>` immediately after a successful bootstrap; `-p` is idempotent (returns the existing pid if `RunAtLoad` already fired, spawns otherwise), making the immediate-start path deterministic while remaining a no-op on the happy path. The kickstart is best-effort (logged, not fatal) — the socket-ping readiness poll stays the authoritative gate.
- **Browser-bridge restart reliably rebinds its fixed ports** ([#990](https://github.com/rust-works/omni-dev/issues/990)): a daemon- or tray-initiated `BridgeService::restart` tears the old planes down and rebinds the *same* fixed ports (default `9998`/`9999`), but a just-closed connection could leave a port in `TIME_WAIT`, so `BridgeServer::start` failed with `EADDRINUSE` — leaving the bridge wedged not-running with no automatic recovery (`status`/`menu` reported not-running until another restart). The bridge now binds its loopback control and WebSocket listeners with `SO_REUSEADDR`, so the rebind succeeds immediately despite a lingering `TIME_WAIT`; standalone `omni-dev browser bridge serve` restarts benefit too. Fail-closed against a *live* listener on the port is unchanged (`SO_REUSEADDR` ≠ `SO_REUSEPORT`), and the loopback-only, token-authenticated trust boundary (ADR-0036) is untouched.
- **launchd plist no longer breaks on paths containing XML metacharacters** ([#991](https://github.com/rust-works/omni-dev/issues/991)): `daemon start` rendered the LaunchAgent plist by interpolating the executable path (`current_exe()`) and the `--socket` path straight into `<string>…</string>`. `&`, `<`, and `>` are all legal in filenames (e.g. a home or socket dir named `A&B`), so such a path produced malformed plist XML and `launchctl bootstrap` failed with an opaque parse error. The two paths are now XML-escaped before interpolation. A non-UTF-8 path — previously corrupted silently via `Path::display()` — is now rejected with a clear error instead, since it cannot be faithfully represented in the UTF-8 plist.
- **Daemon control socket bounds NDJSON line length (local memory-exhaustion DoS)** ([#989](https://github.com/rust-works/omni-dev/issues/989)): the control-socket framing used `LinesCodec::new()`, whose `max_length` defaults to `usize::MAX`, so any local process that connected and streamed bytes **with no newline** grew the per-connection read buffer without bound — and since each accepted connection is a detached `tokio::spawn`, many such connections could run in parallel. The codec is now capped at 1 MiB per line (`MAX_LINE_BYTES`) on both the server and the thin client; an over-limit request gets a single `DaemonReply::err` and the connection is closed (rather than looping forever in the codec's post-overflow discard mode). Found during the post-merge review of #988; see [ADR-0039](docs/adrs/adr-0039.md).
- **Daemon shutdown no longer makes stray pings hang, and `stop`/`restart` honour their ~5s budget** ([#993](https://github.com/rust-works/omni-dev/issues/993)): after the accept loop broke on shutdown, the listener fd stayed open until process exit, so a `connect`+`ping` arriving during `shutdown_all().await` connected at the kernel level but was never accepted — it blocked the caller until the (potentially slow) drain finished. The control socket is now closed **and unlinked before** draining, so such connects fail fast (`ECONNREFUSED`); unlinking up-front also closes a restart race where a replacement daemon could reclaim the stale socket and rebind its own listener mid-drain only to have the old daemon's late unlink delete it. Separately, `wait_until_ready`/`wait_until_down` polled `50×100 ms` with **no per-attempt timeout**, so a single hung `ping` could blow the documented "~5s" budget and couple `stop`/`restart` latency to drain time; each `ping` is now individually bounded (250 ms) and the poll loop enforces a real 5s deadline.
- **`daemon restart` no longer risks leaving the daemon stopped *and* unregistered from launchd** ([#994](https://github.com/rust-works/omni-dev/issues/994)): on macOS, `restart` booted the launchd agent out (`launchd::unload`) *before* re-bootstrapping it via `install_and_load` — but `install_and_load` already boots out any prior agent first, so the explicit pre-bootout was redundant and merely widened a failure window: if the process died or `launchctl bootstrap` failed between that bootout and a successful re-bootstrap, the daemon was left both stopped and unregistered from auto-start (worse than before `restart` ran). The redundant pre-bootout is removed, so the unavoidable bootout→bootstrap gap is now the tight, retrying sequence inside `install_and_load`. This also fixes a second instance of the same bug, where a slow shutdown could make `wait_until_down`'s timeout bail *after* the agent had already been booted out. `stop` still boots out via `unload` to disable auto-start.
- **Confluence attachment upload returned HTTP 405** ([#1005](https://github.com/rust-works/omni-dev/issues/1005)): `confluence attachment upload` (and the `confluence_attachment_upload` MCP tool) POSTed to the Confluence Cloud v2 path `/wiki/api/v2/pages/{id}/attachments`, which has no attachment-creation handler and returns `405 METHOD_NOT_ALLOWED`. Attachment creation is v1-only; uploads now use `POST /wiki/rest/api/content/{id}/child/attachment` and parse the v1 response shape, so the returned attachment metadata (media type, file size, download URL, page id, file id, version) is populated. Listing and deletion (already on supported v2 endpoints) are unchanged.
- **Large (multi-chunk) Snowflake results decode correctly** ([#999](https://github.com/rust-works/omni-dev/issues/999)): the clean-room client parsed each downloaded result chunk as a self-contained JSON array, but Snowflake serves chunks as **bare comma-separated row arrays** (`[r1],[r2],…`) meant to be concatenated — so any query whose result spanned a multi-row chunk failed with `invalid result chunk JSON: trailing characters`. Chunk bytes are now framed with `[` … `]` before parsing (matching the reference connector), so heavy queries returning large result sets decode correctly. (Found via the #999 field test once the inner error was surfaced.)
- **Daemon errors surface the underlying cause** ([#999](https://github.com/rust-works/omni-dev/issues/999)): a control-socket reply collapsed any handler error to its top-level wrapper (e.g. just `Snowflake query failed`), hiding the real reason. Replies now carry the full error chain (`{:#}`), so the actual Snowflake server error code/message — or a client-side polling-deadline timeout (`query did not finish within …`) — reaches the caller, and the engine also logs the inner cause server-side. (`SNOWFLAKE_QUERY_TIMEOUT`, default 3600s, bounds the async-result polling for long-running queries.)
- **Menu-bar tray no longer closes on refresh** ([#999](https://github.com/rust-works/omni-dev/issues/999)): the daemon tray rebuilt its whole menu (`set_menu`) whenever any item's text changed — and on macOS that closes an open menu, so a Snowflake submenu showing live per-session activity would snap shut roughly once a second. The tray now updates item text/enabled **in place** and only rebuilds when the menu *structure* changes (a session added or removed), so an open menu stays open while its live state refreshes. Each authenticated session now also shows **what it's doing** — the running query (a truncated preview) and how long it's been running, or its idle time and query count — across the tray submenu, `daemon status`, and `omni-dev snowflake sessions`.
- **`jira dev` auto-discovery for Bitbucket Server** ([#924](https://github.com/rust-works/omni-dev/issues/924)): provider auto-discovery now queries the DevStatus detail endpoint with the instance-type identifier (e.g. `applicationType=stash`) reported by the summary's `byInstanceType` map, rather than the human-readable display name (`Bitbucket Server`). Previously, any provider whose display name differed from its identifier — Bitbucket Server being the canonical case — was missed, returning empty results.
- **Inline `code` in JFM headings no longer fails the ADF mark validator** ([#1005](https://github.com/rust-works/omni-dev/issues/1005)): A heading authored with backticks — e.g. `` ### `GET /api/services/example` `` — previously produced a `heading` node carrying a `code` mark, which ADF's heading content model forbids, so the document was rejected at write time with an opaque JSONPath-only error. The JFM→ADF converter now strips the `code` mark from heading content (keeping the text as plain — the safe lossy direction, since Atlassian renders no inline-code styling on headings) and emits a warning naming the affected heading. The validator remains the safety net for any other path that produces a code-marked heading.

## [0.28.0] - 2026-06-02

### Added
- **Browser WebSocket bridge** ([#902](https://github.com/rust-works/omni-dev/issues/902)): A new `omni-dev browser` command tree drives HTTP requests **through an authenticated browser tab**, borrowing the browser's SSO/OAuth/cookie session without exfiltrating credentials (a confused-deputy by design). `omni-dev browser bridge serve` runs a long-lived local process exposing two planes — a WebSocket plane the browser connects to via a pasted DevTools snippet (default `127.0.0.1:9999`), and an HTTP control plane the operator drives (default `127.0.0.1:9998`) — joined by an `id`-keyed correlator. Requests reach the page origin via a transparent proxy (any non-`/__bridge/` path), `GET /__bridge/status`, or `POST /__bridge/request`; `omni-dev browser bridge request --url … --method …` is a thin client that injects auth headers itself. The security model is core (see [ADR-0036](docs/adrs/adr-0036.md)): both planes are authenticated and default-closed, with a startup-generated session token never read from argv, control-plane `Host` allowlist + `Origin`/`Sec-Fetch-Site` rejection + mandatory `X-Omni-Bridge: 1` header + no CORS (anti-CSRF / anti-DNS-rebinding), WS token subprotocol, server-side relative-URL-only outbound scope (`--allow-origin` to widen), per-request `504` timeout, and bounded body size/concurrency. Both ports accept `0` to bind an OS-assigned random free port. New modules under `src/browser/` and `src/cli/browser/`; new dependencies `axum`, `tokio-tungstenite`, `rand`. Documented in [docs/browser-bridge.md](docs/browser-bridge.md).

- **Binary Response Bodies in the Browser Bridge** ([#906](https://github.com/rust-works/omni-dev/issues/906)): The browser bridge now round-trips **non-text** response bodies (images, gzipped/encoded blobs, file downloads) intact, closing the v1 gap where the snippet read every response via `resp.text()` and corrupted anything that wasn't UTF-8 text. The pasted snippet now inspects the response `Content-Type`: bodies that aren't `text/*`, JSON, XML, or JavaScript (and bodies with no type) are read via `arrayBuffer()`, base64-encoded, and tagged with `"encoding": "base64"` on the WebSocket frame; text/JSON bodies are unchanged and omit `encoding` (back-compat). The wire protocol gains an optional `encoding` field on `BrowserReply` / `ResponseEnvelope` (`src/browser/protocol.rs`) defaulting to plain text. The transparent proxy (`src/browser/bridge.rs`) base64-decodes before writing the HTTP response body so `curl` receives the raw bytes; `POST /__bridge/request` returns the envelope as-is for the caller to decode. `--max-body-bytes` is now accounted against the **decoded** size, and a malformed or unsupported encoding fails closed with `502`. Covered by new protocol unit tests, a snippet-content assertion, and three integration tests (proxy decodes to raw bytes, request endpoint returns the base64 envelope, oversized decoded body rejected). See [docs/browser-bridge.md](docs/browser-bridge.md) and [ADR-0036](docs/adrs/adr-0036.md).

- **Streaming / Chunked Response Bodies in the Browser Bridge** ([#907](https://github.com/rust-works/omni-dev/issues/907)): The browser bridge can now consume **streaming / long-lived** endpoints (Grafana Live, Server-Sent Events, chunked APIs) that never deliver a final buffered body, closing the v1 gap where every response was buffered under `--request-timeout` (so streaming endpoints timed out with `504` or returned nothing). Streaming is **caller opt-in** — `--stream` on `omni-dev browser bridge request`, `"stream": true` on `POST /__bridge/request`, or `?__stream=1` on the transparent proxy — and buffered (text + binary) behaviour is the unchanged default. The wire protocol (`src/browser/protocol.rs`) gains a `stream` flag on the `Command`, a server→browser `CancelCommand`, a `BrowserFrame` superset that classifies into head / `{seq,chunk}` / `done` / error `StreamItem`s, and an untagged `StreamLine` for NDJSON output. The pasted snippet reads `response.body.getReader()`, emits a head frame then one base64 chunk frame per read then a `done` frame, and cancels its reader on a `cancel` frame. The correlator (`src/browser/bridge.rs`) replaces the single `oneshot` per id with a per-id channel; the transparent proxy streams the decoded bytes as a native chunked HTTP body (curl-friendly) while `POST /__bridge/request` streams an `application/x-ndjson` body (head line, `{seq,chunk}` lines, `{done}` line). For streams, `--request-timeout` is reinterpreted as an **inter-chunk idle timeout** and `--max-body-bytes` as a **cumulative** ceiling; exceeding either limit — or the control-plane consumer disconnecting mid-stream — sends the browser a `cancel` frame so it stops its in-page reader. There is no socket-level backpressure to the browser's reader (memory is bounded by the cumulative cap, an accepted limitation). Covered by new protocol/correlator unit tests, a snippet-content assertion, and integration tests spanning the happy paths (proxy reassembles chunks to a raw body, request endpoint returns ordered NDJSON, `--stream` thin-client round-trip), the limit paths (cumulative cap aborts and cancels, idle timeout ends the stream), and the error/edge paths (no browser → 503, cross-origin → 403, invalid header → 400, error head / chunk-before-head → 502, start idle timeout → 504, undecodable chunk truncates, stray head ignored, unparseable frame survived). See [docs/browser-bridge.md](docs/browser-bridge.md) and [ADR-0036](docs/adrs/adr-0036.md).
- **Per-request `--allow-origin` on the browser bridge `request`** ([#918](https://github.com/rust-works/omni-dev/issues/918)): `omni-dev browser bridge request` gains a `--allow-origin <URL>` flag carrying a **request-scoped** outbound-origin override, so a single request can target a cross-origin URL **without** restarting `serve` and **without** breaking the extension's WebSocket connection. Previously cross-origin was gated only by the `serve --allow-origin` global, which feeds two checks at different times — `ws_origin_allowed` at WebSocket **upgrade** (the extension's `Origin`) and `validate_outbound_url` **per request** (the target URL's origin) — so widening it to permit a cross-origin target simultaneously caused the connection-time check to reject the page's own tab. The new override reaches **only** `validate_outbound_url`, never `ws_origin_allowed`: it takes precedence over the `serve` global for that request's outbound-URL check and leaves the WS upgrade gate untouched. The wire protocol (`src/browser/protocol.rs`) gains an optional `allow_origin` field on `ControlRequest` (`#[serde(default, skip_serializing_if = "Option::is_none")]`, so pre-feature clients are byte-identical on the wire); `dispatch` and `start_stream` (`src/browser/bridge.rs`) resolve the per-request value over the global and emit a WARN at the dispatch site whenever the override is exercised. The override is **per-request and explicit — never a default**: blast radius is bounded by the browser's own CORS (response body readable only if the target sends permissive CORS), and a token holder can already issue authenticated same-origin requests. Covered by protocol round-trip/back-compat tests and bridge unit tests (override wins over and falls back to the global; the WS gate is unaffected). See [docs/browser-bridge.md](docs/browser-bridge.md) and [ADR-0036](docs/adrs/adr-0036.md).
- **Multi-tab routing on the browser bridge via `X-Omni-Bridge-Target`** ([#908](https://github.com/rust-works/omni-dev/issues/908)): The bridge now accepts **multiple authenticated browser tabs concurrently**, replacing the single `Option<WsConn>` connection slot with a `conn_id`-keyed `HashMap<u64, WsConn>` registry (`src/browser/bridge.rs`); each tab authenticates independently via the token subprotocol and a new connection never evicts an existing one. A request selects its target tab by connection id or by unique `Origin` through the new `X-Omni-Bridge-Target` header (or the `target` body field on `POST /__bridge/request`, with the header taking precedence and being stripped before forwarding), surfaced on the thin client as `omni-dev browser bridge request --target <ID|ORIGIN>`. `resolve_target()` routes the request, rejecting ambiguous matches with `409` and unknown targets with `404`; with a single tab connected the target is optional (v1 back-compat) and only becomes required once more than one tab is connected. The protocol (`src/browser/protocol.rs`) gains a `TabInfo { id, origin }` struct and `StatusResponse` grows a `tabs: Vec<TabInfo>` listing every connected tab (`browser_origin` retained for back-compat). See [docs/browser-bridge.md](docs/browser-bridge.md) and [ADR-0036](docs/adrs/adr-0036.md).
- **Per-request `--credentials` on the browser bridge `request`** ([#920](https://github.com/rust-works/omni-dev/issues/920)): `omni-dev browser bridge request` gains a `--credentials <include|omit|same-origin>` flag controlling the browser `fetch()` credentials mode per request. A new `Credentials` enum (`ValueEnum` for clap, with an `as_fetch_value()` wire helper) threads an optional `credentials` field through `ControlRequest` and `Command` (`#[serde(skip_serializing_if)]`, so v1 clients stay byte-identical on the wire) into `dispatch` and `start_stream`; the `browser-bridge.js` snippet reads `cmd.credentials || 'include'` instead of the hard-coded `'include'`. The default stays `include` (send cookies/auth) so existing callers are unaffected. The primary use case is reading wildcard-CORS (`Access-Control-Allow-Origin: *`) cross-origin assets the browser refuses to expose to a credentialed request — pair `--credentials omit` with `--allow-origin`. Covered by unit tests for defaults, all three modes, invalid-value rejection, wire mapping, serialization round-trips, and field omission. See [docs/browser-bridge.md](docs/browser-bridge.md).

- **Actionable Over-`--max-body-bytes` Error + Canonical Pagination Loop** ([#909](https://github.com/rust-works/omni-dev/issues/909)): When a browser response exceeds `--max-body-bytes` (default 8 MiB), the bridge's `502` now names the observed decoded size and the configured limit and steers the operator toward both remedies — page the request to fetch less per call (narrow the time range or lower a `limit`/page size) or raise `--max-body-bytes` — instead of the bare `"browser response body exceeds --max-body-bytes"`. The message is produced in `dispatch()` (`src/browser/bridge.rs`) where the decoded length and configured cap are already in scope, and reaches the `omni-dev browser bridge request` user verbatim via its `bridge returned {status}: {text}` relay. The body cap and its `502` status are unchanged — this is ergonomics, not a loosening of the bound. [docs/browser-bridge.md](docs/browser-bridge.md) gains a runnable canonical drain loop (the Grafana/Loki `direction=backward` worked example), using the `request` thin client + `jq` to take the oldest returned timestamp as the next `end` and repeat until a page is empty, with a note tying a tripped over-limit error back to lowering the page size or raising the cap. No new CLI surface — pagination conventions are per-API, so the bridge stays a thin generic proxy and orchestration is left to the caller. Covered by extending the existing `oversized_decoded_binary_body_is_rejected` integration test to assert the new guidance text.

- **ADF Validator: Quantifier / Arity Checks** ([#733](https://github.com/rust-works/omni-dev/issues/733)): `validate_document` now flags arity violations alongside the existing disallowed-child checks. `mediaSingle` with two `media` children, an empty `bulletList`, an empty `mediaGroup`, an empty `tableRow`, and a `layoutSection` with the wrong number of columns (must be 2 or 3) are all reported as `AdfSchemaViolation::Arity { parent_type, atoms, expected, actual, path }`. The `expected` field carries a new `Quantifier` enum (`ZeroOrOne`, `ZeroOrMore`, `OneOrMore`, `Exactly(n)`, `Range(min, max)`); `path` points at the parent whose count is wrong (whereas `DisallowedChild`'s `path` points at the child). Disallowed children do **not** count toward parent arity — an empty panel containing only an `expand` produces both a `DisallowedChild` (for the expand) and an `Arity` (for the panel needing ≥1 valid child). `unsupportedBlock` / `unsupportedInline` continue to be accepted under any known parent **and** count toward the parent's current term arity, so round-tripped documents carrying preservation wrappers still satisfy `+` / `Exactly(n)` requirements. Two upstream rules are intentionally relaxed for compatibility with real-world inputs: `doc` is `block*` instead of `block+` (so `AdfDocument::new()` — the canonical empty value for missing JIRA descriptions — does not trigger an arity violation) and `tableCell` / `tableHeader` are `block*` instead of `block+` (visibly-empty cells in real Confluence tables are common and accepted by the renderer). The `confluence write --dry-run` output and the HTTP-500 diagnosis in `confluence_write_error` both pick up the new variant via the existing `Display` impl, with arity-specific wording (`'bulletList' must contain at least one 'listItem' (found 0)`). Per-arity hints (empty list, two-media mediaSingle, layoutSection out-of-range, …) are added to the new `adf_hints` module. New module: `src/atlassian/adf_hints.rs`. Modified: `src/atlassian/adf_schema.rs` (full content-model rewrite + new `Quantifier` / `ContentTerm` types + new walker), `src/atlassian/error.rs` (Display switches per variant), `src/atlassian/confluence_api.rs` (uses `adf_hints::hint_for(&violation)`). Covered by 41 unit tests in `adf_schema` plus 6 unit tests in `adf_hints` plus 6 integration tests in `tests/adf_schema_test.rs`.

Covered by 130+ unit tests in the history module including `insta` snapshots pinning the rendered shape of one of each event type both with and without `--exclude-system`, plus end-to-end tests for both-formats-side-by-side, prune scoping per format, partial-jsonl-prefix tolerance, and markdown idempotency on re-run.

- **CommonMark Code Span Handling** ([#578](https://github.com/rust-works/omni-dev/issues/578)): `convert to-adf` no longer splits code-marked text containing backtick characters. Code-span rendering now picks the minimum backtick delimiter length that avoids collisions and applies the CommonMark space-padding rule; the parser matches multi-backtick delimiters with exact-length matching and strips padding spaces symmetrically
- **Integer vs Float Table Width** ([#577](https://github.com/rust-works/omni-dev/issues/577)): ADF→JFM→ADF round-trips no longer coerce integer table width values to floats. A shared `parse_numeric_attr`/`fmt_numeric_attr` helper pair preserves integer-vs-float JSON type based on whether the source had a decimal point
- **Integer/Float in Numeric Attributes** ([#555](https://github.com/rust-works/omni-dev/issues/555)): `layoutColumn`, `mediaSingle`, and file-attachment `media` width/height/`mediaWidth` attributes now preserve the original integer-vs-float JSON number type on round-trip, preventing byte-level JSON inequalities and potential schema validation failures
- **Combined Emoji ShortNames** ([#576](https://github.com/rust-works/omni-dev/issues/576)): Emoji nodes with combined shortNames like `:slightly_smiling_face::bow:` are no longer split into multiple nodes during ADF→JFM→ADF round-trips. The inline parser now extends the match through adjacent `:name:` shortcodes before checking for a trailing attribute directive
- **Pipe Characters in Table Cells** ([#579](https://github.com/rust-works/omni-dev/issues/579)): Literal `|` characters inside pipe-table cell content (including inside inline code spans) are now escaped as `\|` on render and parsed as literal pipes on input, preventing table cells from being split at non-separator pipes
- **Code-Block & Unicode Emoji Shortcodes** ([#552](https://github.com/rust-works/omni-dev/issues/552)): Two ADF↔Markdown round-trip bugs fixed: hardBreak continuation no longer consumes 2-space-indented fenced code blocks or `:::` container directives as continuation lines, and emoji shortcode escaping now uses Unicode `is_alphanumeric` (matching the parser) rather than ASCII-only, so patterns like `:Café:` and `:配置:` are escaped and survive round-trips intact
- **Card URL Brackets** ([#553](https://github.com/rust-works/omni-dev/issues/553)): `::card[...]` / `:card[...]` directives now round-trip correctly even when the URL contains unbalanced `]` or newline characters. URLs that cannot be safely embedded in bracket content are emitted using the quoted `url` attribute form. Additionally, bare URLs inside emphasis, strikethrough, link labels, bracketed spans, or span directives are no longer incorrectly promoted to `inlineCard` nodes (which was silently dropping the enclosing marks)
- **Spurious order=1 on orderedList** ([#547](https://github.com/rust-works/omni-dev/issues/547)): Converting an ADF `orderedList` with no attrs through JFM and back no longer adds a spurious `{"order": 1}` attrs field to the node
- **ADF Mark Ordering** ([#549](https://github.com/rust-works/omni-dev/issues/549)): Inline mark ordering is now preserved across ADF→JFM→ADF round-trips. Markdown wrappers are emitted in mark-array order (outermost first) rather than a fixed priority, so every permutation of `em`, `strong`, `strike`, `underline`, `link`, `annotation`, `textColor`, `backgroundColor`, and `subsup` survives with its original ordering intact
- **Media `occurrenceKey` & Attribute Quoting** ([#550](https://github.com/rust-works/omni-dev/issues/550)): The `occurrenceKey` attribute on `mediaSingle` and `mediaInline` nodes is now preserved on round-trip. Attribute values containing spaces, quotes, closing braces, or backslashes are now quoted and escaped correctly, so arbitrary `id`, `collection`, `url`, `alt`, `localId`, and `occurrenceKey` values round-trip losslessly
- **Annotation-Link Round-Trip Test URL** ([#574](https://github.com/rust-works/omni-dev/pull/574)): Correct the URL used in the annotation-link round-trip test (typo fix)
- **Confluence User Search** ([#542](https://github.com/rust-works/omni-dev/issues/542)): `confluence user search` no longer fails with `missing field 'accountId'` deserialization errors. The response parser now reads the nested `user` object returned by `/wiki/rest/api/search/user` and tolerates user records (such as app users or deactivated users) that omit `accountId`.
- **Empty Task Checkboxes** ([#548](https://github.com/rust-works/omni-dev/issues/548)): Recognise `- [ ]` and `- [x]` markdown task markers even when the checkbox is not followed by a trailing space. Fixes ADF round-trip drift where empty `taskItem` nodes were parsed as `listItem` nodes containing literal `[ ]` text
- **Literal Checkbox Text in Bullet Lists** ([#548](https://github.com/rust-works/omni-dev/issues/548)): Escape the leading `[` when rendering a `bulletList` item whose literal text begins with a sequence that looks like a task checkbox marker (`[ ]`, `[x]`, or `[X]` followed by space, newline, or end), preventing `to-adf` from falsely promoting these bullet items to `taskList`/`taskItem` on round-trip
- **ADF Round-Trip URL Brackets** ([#551](https://github.com/rust-works/omni-dev/issues/551)): Preserve square brackets in URLs embedded in link-marked text so ADF→JFM→ADF round-trips no longer leak `\[`/`\]` escapes or split the text into corrupted `inlineCard` nodes
- **ADF Mark Combinations** ([#554](https://github.com/rust-works/omni-dev/issues/554)): `textColor`, `backgroundColor`, `subsup`, `underline`, and `annotation` marks were silently dropped when combined with a `code` mark or with each other. Marks are now preserved by nesting `:span[…]{…}` and `[…]{…}` wrappers based on the original mark order
- **Underscore boundary in span directives** ([#554](https://github.com/rust-works/omni-dev/issues/554)): `from-adf` now escapes underscores at text-node boundaries (per the CommonMark intraword rule). Previously, a plain text node ending with `_ ` followed by a textColor span whose text started with `_` produced JFM that the parser saw as italic, destroying the span directive and losing the textColor mark
